БЛОК 1. PYTHON — ТЕОРИЯ

1►Вопрос 1. Структура программы на Python. Функции ввода/вывода (print, input)

Структура программы. Программа на Python состоит из последовательности инструкций, которые интерпретатор выполняет сверху вниз. Типичные элементы: импорт модулей, объявление функций и классов, основной код. Отступы (4 пробела) задают вложенность блоков — фигурных скобок нет. Часто точка входа оформляется через if __name__ == "__main__":.

import math                     # импорт модуля

def square(x):                  # определение функции
    return x * x

if __name__ == "__main__":      # точка входа
    print(square(5))

Вывод — print(). Выводит данные в консоль. Аргументы разделяются sep (по умолчанию пробел), в конце добавляется end (по умолчанию \n).

print("Привет", "мир")            # Привет мир
print("a", "b", sep="-")          # a-b
print("без переноса", end="")     # без перехода на новую строку
print(f"Сумма = {2 + 3}")         # f-строка: Сумма = 5

Ввод — input(). Считывает строку с клавиатуры (всегда тип str). Для чисел нужно явное приведение.

name = input("Имя: ")            # строка
age = int(input("Возраст: "))    # целое
price = float(input("Цена: "))   # вещественное
print(f"{name}, через год вам {age + 1}")
2►Вопрос 2. Арифметические и логические операции

Арифметические:

Оператор  |  Действие  |  Пример  |  Результат
+ - *  |  сложение, вычитание, умножение  |  2 * 3  |  6
/  |  деление (всегда float)  |  7 / 2  |  3.5
//  |  целочисленное деление  |  7 // 2  |  3
%  |  остаток от деления  |  7 % 2  |  1
**  |  возведение в степень  |  2 ** 10  |  1024

Логические: and, or, not — работают с булевыми значениями, возвращают True/False.

a, b = 5, 0
print(a > 3 and b == 0)   # True
print(a < 3 or b == 0)    # True
print(not (a > 3))        # False
3►Вопрос 3. Операции сравнения

Операторы ==, !=, >, <, >=, <= возвращают bool. Сравнивать можно числа, строки (лексикографически), логические значения. Поддерживаются цепочки сравнений.

print(5 == 5.0)          # True (значения равны)
print("abc" < "abd")     # True (лексикографически)
print(True == 1)         # True (True эквивалентно 1)
print(1 < 3 < 5)         # True (цепочка: 1<3 и 3<5)
print(3 != 4)            # True
4►Вопрос 4. Конструкция if (полная и неполная форма)

if управляет ветвлением. Неполная форма — только if. Полная — с else и дополнительными ветвями elif.

n = 5
if n > 0:
    print("Положительное")

if n > 0:
    print("Положительное")
elif n == 0:
    print("Ноль")
else:
    print("Отрицательное")

Синтаксис: после условия — двоеточие, тело с отступом.

5►Вопрос 5. Цикл while

Цикл с предусловием: повторяет тело, пока условие истинно.

i = 1
while i <= 5:
    print(i)
    i += 1          # обязательно меняем счётчик, иначе вечный цикл

s = 0
x = int(input())
while x != 0:
    s += x
    x = int(input())
print("Сумма:", s)
6►Вопрос 6. Цикл for

Перебирает элементы любой последовательности (списка, строки, кортежа, range).

for c in "Hello":          # по символам строки
    print(c)

for x in [10, 20, 30]:     # по элементам списка
    print(x)

for i in range(5):         # 0,1,2,3,4
    print(i)

for idx, val in enumerate(["a", "b"]):   # с индексом
    print(idx, val)

Особенность: for не требует ручного счётчика — итерация по объекту идёт автоматически.

7►Вопрос 7. Вложенный цикл

Цикл внутри другого цикла. Применяют для обработки таблиц (двумерных структур), комбинаций пар, матриц. Уровень вложенности — сколько циклов друг в друге.

for i in range(1, 4):          # внешний цикл
    for j in range(1, 4):      # внутренний цикл
        print(f"{i}*{j}={i*j}", end="  ")
    print()                    # перенос строки после строки таблицы

На каждую итерацию внешнего цикла внутренний выполняется полностью.

8►Вопрос 8. Функция range()

Генерирует последовательность целых чисел.

range(5)         # 0,1,2,3,4              (1 аргумент: stop)
range(2, 6)      # 2,3,4,5                (2 аргумента: start, stop)
range(1, 10, 2)  # 1,3,5,7,9             (3 аргумента: start, stop, step)
range(10, 0, -1) # 10,9,...,1            (обратный шаг)

print(list(range(0, 10, 3)))   # [0, 3, 6, 9]

Правая граница stop не включается.

9►Вопрос 9. Строки. Операции, функции, срезы

Строка (str) — неизменяемая упорядоченная последовательность символов.

s = "Python"
print(len(s))          # 6 — длина
print(s + "3")         # конкатенация: Python3
print(s * 2)           # повтор: PythonPython
print("y" in s)        # True — вхождение

Функции/методы: len(), s.upper(), s.lower(), s.find(), s.replace(), s.split(), s.strip(), s.count().

Срез s[start:stop:step] — подстрока (stop не включается):

print(s[0:3])    # Pyt
print(s[2:])     # thon
print(s[:4])     # Pyth
print(s[::-1])   # nohtyP (разворот)
print(s[-1])     # n (последний символ)
10►Вопрос 10. Кортеж (tuple)

Кортеж — неизменяемая упорядоченная последовательность.

t = (1, 2, 3)            # создание
t2 = 4, 5, 6            # скобки необязательны
t3 = (7,)              # кортеж из одного элемента (запятая обязательна!)
empty = ()             # пустой

print(t[0])            # доступ по индексу: 1
print(t[-1])           # 3
print(t[0:2])          # срез: (1, 2)
for x in t:            # перебор
    print(x)

Применяют, когда набор данных не должен меняться (координаты, константы).

11►Вопрос 11. Словарь (dict)

Словарь — изменяемая коллекция пар «ключ: значение». Ключи уникальны.

d = {"name": "Anna", "age": 20}   # создание
d["city"] = "Aktobe"              # добавление
print(d["name"])                  # доступ: Anna

Методы: keys(), values(), items(), get(), pop(), update(), clear().

print(d.keys())          # dict_keys(['name','age','city'])
print(d.values())        # значения
print(d.get("age"))      # 20; get не падает, если ключа нет
d.pop("age")             # удаляет ключ и возвращает значение
for k, v in d.items():   # перебор пар
    print(k, v)
12►Вопрос 12. Пользовательская функция

Объявляется через def, может принимать параметры и возвращать значение (return). Особенности Python: значения по умолчанию, именованные аргументы, args/*kwargs, возврат нескольких значений.

def greet(name, greeting="Привет"):   # параметр со значением по умолчанию
    return f"{greeting}, {name}!"

print(greet("Иван"))                  # Привет, Иван!
print(greet("Иван", greeting="Hi"))   # именованный аргумент

def min_max(lst):                     # возврат нескольких значений
    return min(lst), max(lst)

lo, hi = min_max([3, 1, 7])           # 1 7
13►Вопрос 13. Файлы в Python
f = open("data.txt", "r", encoding="utf-8")   # синтаксис открытия

Режимы: r (чтение), w (запись с очисткой), a (дозапись), r+ (чтение+запись), b (бинарный).

Атрибуты объекта файла: f.name, f.mode, f.closed, f.encoding.

Методы: read(), readline(), readlines(), write(), writelines(), close(), seek(), tell().

with open("data.txt", "w", encoding="utf-8") as f:  # with закрывает файл автоматически
    f.write("Первая строка\n")

with open("data.txt", "r", encoding="utf-8") as f:
    for line in f:
        print(line.strip())
14►Вопрос 14. GUI. Модуль Tkinter, стандартные виджеты

Tkinter — встроенная в Python библиотека для создания графического интерфейса (обёртка над Tk). Позволяет создавать окна, размещать элементы управления, обрабатывать события.

Стандартные виджеты: Label (надпись), Button (кнопка), Entry (однострочное поле), Text (многострочное поле), Checkbutton, Radiobutton, Listbox, Frame, Menu, Canvas, Scrollbar, Scale.

import tkinter as tk

root = tk.Tk()
root.title("Пример")
tk.Label(root, text="Имя:").pack()
entry = tk.Entry(root)
entry.pack()
tk.Button(root, text="OK", command=lambda: print(entry.get())).pack()
root.mainloop()
15►Вопрос 15. Меню в Tkinter

Меню создаётся виджетом Menu. Главное меню привязывается к окну через root.config(menu=...), выпадающие подменю добавляются через add_cascade, пункты — add_command, разделитель — add_separator.

import tkinter as tk

root = tk.Tk()
menubar = tk.Menu(root)

file_menu = tk.Menu(menubar, tearoff=0)
file_menu.add_command(label="Открыть", command=lambda: print("Открыть"))
file_menu.add_separator()
file_menu.add_command(label="Выход", command=root.quit)
menubar.add_cascade(label="Файл", menu=file_menu)

root.config(menu=menubar)
root.mainloop()
16►Вопрос 16. Списки. Простые и вложенные

Список (list) — изменяемая упорядоченная коллекция.

a = [1, 2, 3]                 # создание
a = list(range(5))            # из range
print(a[0])                   # доступ: 0
a.append(99)                  # добавление
a[1] = 50                     # изменение (список изменяем!)

m = [[1, 2, 3], [4, 5, 6]]
print(m[0][1])                # 2 — строка 0, столбец 1
for row in m:                 # перебор вложенного списка
    for x in row:
        print(x, end=" ")
    print()

Особенность вложенных списков: элемент сам является списком, доступ — двойной индекс m[i][j].

17►Вопрос 17. Операторы break и continue
  • break — немедленно прерывает цикл целиком.
  • continue — пропускает остаток текущей итерации и переходит к следующей.
for i in range(1, 6):
    if i == 3:
        break          # прервётся на 3
    print(i)           # 1 2

for i in range(1, 6):
    if i == 3:
        continue       # пропустит 3
    print(i)           # 1 2 4 5
18►Вопрос 18. Множество (set)

Множество — изменяемая неупорядоченная коллекция уникальных элементов.

s = {1, 2, 3, 3}        # {1, 2, 3} — дубликаты убираются
s2 = set([1, 2, 2])     # {1, 2}
empty = set()           # пустое ({} — это словарь!)

s.add(5)                # добавление
s.discard(1)            # удаление
print(2 in s)           # True — проверка вхождения (нет доступа по индексу!)

a, b = {1, 2, 3}, {2, 3, 4}
print(a | b)            # объединение {1,2,3,4}
print(a & b)            # пересечение {2,3}
print(a - b)            # разность {1}

К элементам обращаются перебором или проверкой in — индексации нет.

21►Вопрос 21. Модули. import vs from ... import

Модуль — файл .py с функциями/классами для повторного использования.

import math                  # импорт всего модуля
print(math.sqrt(16))         # обращение через имя модуля

from math import sqrt, pi    # импорт конкретных имён
print(sqrt(16))              # без префикса

from math import *           # всё (не рекомендуется — засоряет пространство имён)
import math as m             # псевдоним

Свой модуль: создаём mytools.py с функциями, затем import mytools и mytools.func().

Разница: import math требует префикса math., from math import sqrt даёт прямой доступ к sqrt.

22►Вопрос 22. Рекурсивные функции

Рекурсия — функция вызывает саму себя. Обязательны: базовый случай (условие остановки) и шаг рекурсии (приближение к базовому случаю). Без базового случая — бесконечная рекурсия и переполнение стека.

def factorial(n):
    if n <= 1:           # базовый случай
        return 1
    return n * factorial(n - 1)   # шаг рекурсии

print(factorial(5))      # 120

На что обратить внимание: наличие условия выхода, уменьшение аргумента, глубина рекурсии (по умолчанию ~1000).

1►Вопрос 1. Назначение и функции ИСрПО. Подбор средств для CI

Инструментальные средства разработки ПО (ИСрПО) — это программы, помогающие создавать, тестировать, сопровождать и сопровождать другое ПО. Назначение — повысить производительность разработчика, снизить число ошибок и автоматизировать рутинные операции.

Основные функции: редактирование кода, трансляция (компиляция/интерпретация), отладка, тестирование, сборка, управление версиями, документирование, моделирование.

Подбор средств для непрерывной интеграции (CI):

  • Система контроля версий — Git (GitHub/GitLab) — хранит код и историю изменений.
  • CI/CD-сервер — Jenkins, GitLab CI, GitHub Actions — автоматически собирает и тестирует

  • Сборщик — Maven/Gradle (Java), npm (JS), pip (Python).
  • Средства тестирования — JUnit, pytest, Selenium.
  • Контейнеризация — Docker — единое окружение для сборки и тестов.

Обоснование выбора: связка Git + GitHub Actions + Docker + автотесты обеспечивает, что после каждого изменения код автоматически собирается, прогоняется через тесты и при успехе готов к развёртыванию — это и есть суть непрерывной интеграции: раннее обнаружение ошибок и стабильно работающая основная ветка.

2►Вопрос 2. Методология, метод, средство. Классификация методологий
  • Методология — система принципов, методов и правил организации процесса разработки ПО

  • Метод — конкретный способ выполнения определённого этапа (например, метод декомпозиции,

  • Средство (инструмент) — программа или нотация, реализующая метод (IDE, CASE-средство,

Общая классификация методологий:

  • Тяжёлые (предиктивные/каскадные) — жёсткое планирование, подробная документация

  • Гибкие (Agile) — итеративность, адаптивность, минимум документации (Scrum, Kanban, XP).
  • По подходу к проектированию: структурные (функциональная декомпозиция) и

3►Вопрос 3. Модели ЖЦ ПО. Каскадная vs итеративная

Жизненный цикл (ЖЦ) ПО — период от идеи до снятия продукта с эксплуатации.

Каскадная (водопадная) модель — этапы выполняются строго последовательно (анализ → проектирование → реализация → тестирование → внедрение), возврат к предыдущему этапу затруднён. Подходит для проектов с чёткими, неизменными требованиями.

Итеративная модель — продукт создаётся за несколько повторяющихся циклов (итераций); каждая итерация даёт работающую версию, требования уточняются по ходу.

Критерий  |  Каскадная  |  Итеративная
Порядок этапов  |  строго последовательный  |  повторяющиеся циклы
Требования  |  фиксированы в начале  |  уточняются по ходу
Результат  |  в конце проекта  |  после каждой итерации
Гибкость к изменениям  |  низкая  |  высокая
Риски  |  выявляются поздно  |  выявляются рано
4►Вопрос 4. Инструменты разработки vs инструменты сопровождения
  • Инструменты разработки используются при создании продукта: редакторы кода,

компиляторы, отладчики, IDE, средства проектирования (UML), сборщики. Цель — построить работающее ПО.

  • Инструменты сопровождения используются после выпуска, при эксплуатации: системы учёта

ошибок (баг-трекеры), мониторинг, профилировщики, средства управления версиями и обновлениями, средства резервного копирования. Цель — поддерживать и улучшать уже работающий продукт (исправление ошибок, адаптация, оптимизация).

Разработка создаёт продукт, сопровождение поддерживает его работоспособность и развитие на протяжении эксплуатации.

5►Вопрос 5. Гибкие методологии. Как Scrum и Kanban реализуют гибкий подход

Гибкие методологии (Agile): SCRUM, KANBAN, DSDM, MSF, ALM, XP, RAD — основаны на итеративности, тесном взаимодействии с заказчиком, готовности к изменениям и быстрой поставке работающих версий.

SCRUM реализует гибкость через короткие фиксированные итерации — спринты (1–4 недели). Работа берётся из бэклога продукта, на спринт формируется бэклог спринта. Есть роли (Product Owner, Scrum Master, команда), события (планирование, ежедневный стендап, обзор, ретроспектива). После каждого спринта — работающий инкремент, что позволяет быстро реагировать на изменения требований.

KANBAN реализует гибкость через визуализацию потока работ на доске (столбцы «To Do → In Progress → Done») и ограничение незавершённой работы (WIP-лимиты). Нет фиксированных спринтов — задачи вытягиваются по мере освобождения исполнителей (поточная, непрерывная поставка). Гибкость — в непрерывной приоритизации и быстром прохождении задач через поток.

Оба подхода итеративны и адаптивны, но Scrum — спринтами, Kanban — непрерывным потоком.

6►Вопрос 6. Бизнес-моделирование. Типы моделей. Графическая vs имитационная

Бизнес-моделирование — описание бизнес-процессов организации для их анализа, оптимизации и автоматизации.

Типы моделей бизнес-процессов:

  • Графические — наглядное изображение процесса схемами/диаграммами (блок-схемы, IDEF, UML).
  • Имитационные — модель, которая выполняется (симулируется) во времени и показывает

поведение процесса при разных параметрах.

  • Исполняемые/функциональные — формальное описание, которое можно автоматически

исполнить (BPEL).

  • Информационные — описывают данные и их потоки (ER-модели, DFD).

Графическая vs имитационная: графическая модель статична — это «чертёж» процесса для понимания структуры. Имитационная модель динамична — её запускают, она воспроизводит работу процесса во времени, позволяет оценить производительность, узкие места, влияние изменений до их внедрения.

7►Вопрос 7. Методы моделирования бизнес-процессов. Назначение Role Activity Diagram

Методы: Flow Chart Diagram (блок-схема), Data Flow Diagram (DFD — диаграмма потоков данных), Role Activity Diagram (RAD — диаграмма ролей и действий), IDEF (IDEF0, IDEF3 и др.), UML (Unified Modeling Language).

Role Activity Diagram (RAD) используется для описания бизнес-процесса с точки зрения ролей (участников) и выполняемых ими действий, а также взаимодействия между ролями. Она показывает, кто что делает, в каком порядке, где роли передают работу друг другу и где происходит синхронизация/ветвление. Применяется, когда важно понять распределение обязанностей и взаимодействие участников процесса.

8►Вопрос 8. AllFusion Process Modeler и MS Visio

AllFusion Process Modeler (ранее BPwin) — специализированное CASE-средство для функционального моделирования и анализа бизнес-процессов. Поддерживает нотации IDEF0 (функциональная модель), IDEF3 (модель потоков работ) и DFD. Назначение — построение структурированных моделей деятельности предприятия, анализ и оптимизация процессов.

MS Visio — универсальный редактор деловой графики и диаграмм от Microsoft. Позволяет строить блок-схемы, схемы баз данных, UML-диаграммы, сетевые схемы, организационные диаграммы. Назначение — наглядное визуальное документирование процессов, систем и структур; менее формализован, чем BPwin, но гораздо шире по типам диаграмм.

Различие: AllFusion — строгий инструмент функционального моделирования (анализ), Visio — универсальный инструмент рисования разнообразных схем (визуализация).

9►Вопрос 9. Среды реализации. IDE. Цель использования. Популярность Visual Studio

Интегрированная среда разработки (IDE) объединяет в одной программе все инструменты разработчика: редактор кода, компилятор/интерпретатор, отладчик, средства сборки и др.

Основная цель использования среды разработки — повысить производительность за счёт объединения инструментов в едином интерфейсе: не нужно переключаться между разными программами, всё взаимосвязано (например, ошибка компиляции сразу подсвечивается в редакторе).

Почему Microsoft Visual Studio популярна: мощная и зрелая IDE с поддержкой множества языков (C#, C++, F#, VB и др.); богатый отладчик, IntelliSense (умное автодополнение), интеграция с .NET, системами контроля версий, средствами тестирования и развёртывания; большое сообщество, документация и расширения. Удобна для разработки от настольных до веб- и облачных приложений.

10►Вопрос 10. Виртуальные машины. Принцип работы. VirtualBox vs VMware Workstation

Виртуальная машина (ВМ) — программная эмуляция компьютера, позволяющая запускать внутри одной физической машины другую (гостевую) ОС. Принцип работы: специальная программа — гипервизор — выделяет виртуальные ресурсы (процессор, память, диск, сеть) и изолирует гостевую ОС от хостовой, перехватывая и обрабатывая её обращения к оборудованию.

Критерий  |  VirtualBox  |  VMware Workstation
Разработчик  |  Oracle  |  VMware (Broadcom)
Лицензия  |  бесплатный, open source  |  платный (есть бесплатный Player)
Производительность  |  хорошая  |  выше, особенно с 3D/нагрузкой
Удобство/функции  |  базовые, достаточно для учёбы  |  продвинутые (снимки, клоны, интеграция)
Платформы  |  Windows, Linux, macOS  |  Windows, Linux

VirtualBox — бесплатное универсальное решение для учёбы и личного использования; VMware Workstation — профессиональный платный продукт с большей производительностью и возможностями.

11►Вопрос 11. ВМ: свойства и типы. Польза для тестирования ПО

Свойства ВМ: изоляция (гостевая ОС отделена от хоста), инкапсуляция (вся ВМ — это набор файлов, легко копировать/переносить), независимость от оборудования, снимки состояния (snapshots).

Типы виртуализации/ВМ:

  • Системные ВМ — эмулируют целый компьютер с ОС (VirtualBox, VMware).
  • Программные (процессные) ВМ — выполняют программы определённой среды (JVM для Java,

CLR для .NET).

  • По типу гипервизора: Type 1 (нативный, работает прямо на железе — ESXi, Hyper-V) и

Type 2 (хостовый, поверх ОС — VirtualBox, VMware Workstation).

Польза для тестирования ПО: можно быстро создать чистое тестовое окружение, проверить продукт на разных ОС и конфигурациях без отдельных компьютеров; снимки позволяют откатиться к исходному состоянию после теста; изоляция защищает основную систему от сбоев и вредоносного кода; легко воспроизводить условия и клонировать стенды.

12►Вопрос 12. SDLC. Этапы технологии разработки. Определение

SDLC (Software Development Life Cycle) — жизненный цикл разработки ПО, совокупность этапов от зарождения идеи до вывода продукта из эксплуатации.

Этапы:

  • Анализ требований — сбор и формализация требований заказчика.
  • Проектирование — разработка архитектуры и структуры системы.
  • Реализация (кодирование) — написание программного кода.
  • Тестирование — проверка соответствия требованиям, поиск ошибок.
  • Внедрение (развёртывание) — установка у заказчика, ввод в эксплуатацию.
  • Сопровождение — поддержка, исправление ошибок, доработка.
13►Вопрос 13. ЖЦ ПО. Цель вспомогательных средств поддержки

Вспомогательные средства поддержки ЖЦ — это инструменты, которые сопровождают процесс разработки, но не создают код напрямую. Их цель — обеспечить управление, контроль качества и организацию работы на всех этапах ЖЦ.

К ним относятся: системы управления версиями (Git), системы управления проектами и задачами (Jira, Trello), средства документирования, баг-трекеры, средства автоматизации сборки и тестирования, средства резервного копирования.

Цель — повысить управляемость проекта, сохранить историю изменений, обеспечить взаимодействие команды, контроль качества и прослеживаемость требований на протяжении всего жизненного цикла.

14►Вопрос 14. Спиральная модель ЖЦ

Спиральная модель (предложена Боэмом) сочетает итеративность с акцентом на анализ рисков. Разработка идёт по виткам спирали; каждый виток включает четыре сектора:

  • Определение целей и требований к данному витку.
  • Анализ рисков и их снижение (главная особенность модели).
  • Разработка и тестирование очередной версии (прототипа).
  • Планирование следующего витка и оценка результатов заказчиком.

С каждым витком продукт уточняется и расширяется. Модель подходит для крупных, сложных и рискованных проектов, где требования заранее не полностью ясны. Минус — сложность управления и высокая стоимость анализа рисков.

15►Вопрос 15. Отладка и тестирование: определения
  • Тестирование — процесс выполнения программы с целью обнаружить наличие ошибок и

проверить соответствие требованиям. Тестирование отвечает на вопрос «есть ли ошибка?».

  • Отладка (debugging) — процесс локализации (поиска) и устранения уже обнаруженных

ошибок. Отладка отвечает на вопрос «где ошибка и как её исправить?».

Тестирование выявляет факт ошибки, отладка находит её причину и исправляет.

16►Вопрос 16. Этапы логического проектирования. Декомпозиция задачи

Логическое проектирование — этап, на котором определяется логическая структура будущей программы независимо от конкретной реализации: алгоритмы, структуры данных, модули и связи между ними.

Декомпозиция задачи — разбиение сложной исходной задачи на более простые подзадачи (модули, функции), каждую из которых легче понять, реализовать и протестировать. Применяется принцип «разделяй и властвуй»: задача делится сверху вниз до уровня элементарных операций. Декомпозиция упрощает разработку, повышает читаемость, позволяет распределить работу между исполнителями и повторно использовать модули.

17►Вопрос 17. Программная среда разработки. Взаимодействие инструментов

Программная среда разработки — совокупность взаимосвязанных инструментов для создания ПО. Взаимодействие инструментов: разработчик пишет код в редакторе; компилятор/интерпретатор преобразует код в исполняемую форму, возвращая редактору сообщения об ошибках, которые сразу подсвечиваются; отладчик позволяет пошагово выполнить программу, используя информацию компилятора; сборщик объединяет модули и библиотеки в готовый продукт; система контроля версий фиксирует изменения. Все компоненты обмениваются данными через единую среду, поэтому результат работы одного инструмента сразу доступен другому — это и обеспечивает целостность и скорость процесса.

18►Вопрос 18. IDE и её структура. Классификация. Как облегчает программирование

IDE (Integrated Development Environment) — единая среда, объединяющая инструменты разработки.

Структура (компоненты): редактор кода, компилятор/интерпретатор, отладчик, средства автодополнения (IntelliSense), система сборки, интеграция с контролем версий, менеджер проектов, средства тестирования.

Классификация IDE:

  • по числу языков: одноязычные (Turbo Pascal) и многоязычные (Visual Studio, Eclipse);
  • по платформе: локальные (настольные) и облачные/онлайн (Replit, Cloud9);
  • по специализации: универсальные и предметно-ориентированные (например, для мобильной

разработки — Android Studio).

Как облегчает программирование: объединяет все инструменты в одном окне, ускоряет ввод кода (автодополнение, шаблоны), мгновенно показывает синтаксические ошибки, упрощает отладку и сборку, автоматизирует рутину — это снижает число ошибок и повышает производительность.

19►Вопрос 19. Выбор IDE. Инструменты, полезные при командной разработке

Выбор IDE зависит от языка, платформы, требований проекта, лицензии и привычек команды.

Инструменты IDE, наиболее полезные при командной разработке:

  • Интеграция с системой контроля версий (Git) — совместная работа над кодом, история,

ветвление, слияние.

  • Code review / комментирование — просмотр изменений коллег.
  • Единый стиль кода (форматтеры, линтеры) — согласованность кода в команде.
  • Средства совместной работы (Live Share) — одновременное редактирование.
  • Интеграция с трекерами задач (Jira) и CI/CD — связь кода с задачами и автосборкой.
  • Средства тестирования — общий набор автотестов.

Эти инструменты обеспечивают согласованность, прозрачность и координацию работы участников.

20►Вопрос 20. Обзор сред: JDK, BlackBox, Visual Studio, Eclipse. Роль IDE. Кроссплатформенность Eclipse
  • JDK (Java Development Kit) — комплект разработчика Java (компилятор javac, JVM,

библиотеки, утилиты). Строго говоря, это не IDE, а набор инструментов.

  • BlackBox — среда разработки на языке Component Pascal (Oberon), компактная, учебная.
  • Microsoft Visual Studio — мощная многоязычная IDE (преимущественно .NET, C#, C++).
  • Eclipse — свободная расширяемая IDE, в первую очередь для Java, с плагинами для многих

языков.

Роль IDE в программировании: объединяет редактор, компилятор, отладчик и другие средства, автоматизирует разработку, ускоряет написание и отладку кода, снижает число ошибок.

Почему Eclipse кроссплатформенна: написана на Java и работает на JVM, поэтому запускается на Windows, Linux, macOS без изменения кода; кроме того, имеет модульную (плагинную) архитектуру, позволяющую добавлять поддержку разных языков и платформ.

21►Вопрос 21. Windows SDK. Состав. Назначение

SDK (Software Development Kit) — комплект средств разработки для определённой платформы.

Windows SDK — набор инструментов для создания приложений под ОS Windows.

Основные компоненты Windows SDK:

  • Заголовочные файлы (headers) и библиотеки (libraries) Windows API;
  • Компиляторы и компоновщики (или интеграция с ними);
  • Документация и примеры по Windows API;
  • Утилиты и инструменты — отладчики, профилировщики, средства подписи и упаковки,

редакторы ресурсов.

Назначение SDK при разработке Windows-приложений — предоставить разработчику всё необходимое для обращения к функциям операционной системы (Windows API): создание окон, работа с файлами, графикой, сетью и устройствами. Без SDK программист не имел бы интерфейсов и инструментов для взаимодействия с платформой.

22►Вопрос 22. ВМ: типы и свойства. Зачем в тестировании ПО

Свойства ВМ: изоляция, инкапсуляция (вся ВМ — набор файлов), аппаратная независимость, поддержка снимков (snapshot), возможность клонирования.

Типы: системные (эмулируют компьютер целиком — VirtualBox, VMware) и процессные/программные (JVM, .NET CLR); по гипервизору — Type 1 (нативные) и Type 2 (хостовые).

Зачем в тестировании ПО: ВМ позволяют тестировать программу на разных операционных системах и конфигурациях без покупки нескольких компьютеров; обеспечивают чистое и повторяемое окружение; снимки дают возможность мгновенно вернуть систему в исходное состояние после теста; изоляция защищает основную ОС от сбоев и вирусов; легко создавать и удалять тестовые стенды. Это ускоряет и удешевляет тестирование.

23►Вопрос 23. Инструменты для работы с ВМ. Разница JDK и JVM

Инструменты работы с ВМ: VMware Workstation, Microsoft Virtual PC, Oracle VirtualBox, Parallels Desktop (для macOS). Все они создают и управляют виртуальными машинами на персональном компьютере.

Разница JDK и JVM:

  • JVM (Java Virtual Machine) — виртуальная машина Java, которая исполняет байт-код

Java; обеспечивает кроссплатформенность («написано один раз — работает везде»). Входит в состав среды выполнения JRE.

  • JDK (Java Development Kit) — комплект разработчика: включает компилятор javac,

библиотеки, утилиты и среду выполнения JRE (а значит и JVM).

Кратко: JDK нужен для разработки программ на Java и содержит JVM; JVM нужна для выполнения уже скомпилированных программ. JDK ⊃ JRE ⊃ JVM.

24►Вопрос 24. Правовые методы защиты программных продуктов и баз данных

ПО и базы данных охраняются как объекты интеллектуальной собственности.

Правовые методы защиты:

  • Авторское право — защищает программу как литературное произведение (исходный и объектный

код, базы данных как составные произведения); возникает автоматически при создании.

  • Патентное право — защищает оригинальные технические решения, алгоритмы (в ряде стран).
  • Лицензионные соглашения (EULA) — регулируют условия использования (проприетарные,

свободные/открытые лицензии — GPL, MIT и т. д.).

  • Регистрация программ и баз данных в уполномоченном органе — подтверждает авторство.
  • Коммерческая тайна — защита нераскрытого кода и ноу-хау.
  • Товарный знак — защита названия и бренда продукта.

Эти методы обеспечивают правообладателю исключительные права на использование, распространение и защиту от незаконного копирования.

25►Вопрос 25. Стандартизация в управлении качеством. Основные функции

Стандартизация в управлении качеством ПО — установление единых требований, правил и норм к процессам и продуктам разработки (стандарты ISO/IEC, ГОСТ, ЕСПД).

Основные функции стандартизации:

  • Обеспечение качества — единые критерии оценки продукта и процессов.
  • Унификация и совместимость — согласованность форматов, интерфейсов, документации.
  • Регламентация процессов — описание порядка разработки, тестирования, сопровождения.
  • Обеспечение надёжности и безопасности ПО.
  • Упорядочение документации (например, ЕСПД — единая система программной документации).
  • Повышение взаимозаменяемости и повторного использования компонентов.
  • Основа для сертификации продукта и процессов.
26►Вопрос 26. Этапы физического проектирования ПО

Физическое проектирование — этап, на котором логическая модель преобразуется в конкретную реализацию с учётом выбранной платформы, языка и СУБД.

Основные этапы:

  • Выбор технических и программных средств — язык, СУБД, платформа, архитектура.
  • Проектирование физической структуры данных — таблицы, индексы, типы данных,

размещение на носителях.

  • Проектирование модулей и интерфейсов — конкретная структура программных компонентов.
  • Распределение данных и компонентов (для распределённых систем).
  • Оптимизация — производительность, объём памяти, скорость доступа.
  • Подготовка к реализации — спецификации для кодирования.

В отличие от логического (что делать), физическое проектирование определяет, как это будет реализовано на конкретной платформе.

27►Вопрос 27. Техническое задание. Разделы (пример)

Техническое задание (ТЗ) — основной документ, определяющий требования к разрабатываемому ПО. По ГОСТ 19.201 (ЕСПД) типовые разделы:

  • Введение — наименование, область применения программы.
  • Основания для разработки — документ-основание, заказчик, исполнитель.
  • Назначение разработки — функциональное и эксплуатационное назначение.
  • Требования к программе:
  • требования к функциональным характеристикам;
  • требования к надёжности;
  • условия эксплуатации;
  • требования к составу и параметрам технических средств;

Критерий  |  Тестирование  |  Отладка
Цель  |  обнаружить наличие ошибок  |  найти причину и устранить ошибку

CASE-средства (Computer-Aided Software Engineering) — программы для автоматизации проектирования и разработки ПО (моделирование, генерация кода, документирование).

Основные факторы эффективности CASE-технологии:

  • автоматизация рутинных операций (генерация кода, документации);
  • единая модель проекта и контроль её целостности;
  • раннее выявление ошибок проектирования;
  • повышение качества и сопровождаемости ПО;
  • ускорение разработки и снижение трудозатрат;

30►Вопрос 30. Трансляция, связывание/компоновка, загрузка

Трансляция — преобразование исходного кода в машинный (или промежуточный) код. Два вида:

  • Компиляция — весь исходный код целиком переводится в исполняемый файл заранее

(C, C++); выполняется быстро, но требует пересборки при изменениях.

  • Интерпретация — код выполняется построчно интерпретатором во время работы (Python,

JavaScript); гибче и переносимее, но медленнее.

Тип связывания/компоновки (linking) — объединение объектных модулей и библиотек в исполняемую программу:

  • Статическое связывание — код библиотек включается в исполняемый файл при сборке

(файл больше, но автономен).

  • Динамическое связывание — библиотеки (DLL/.so) подключаются во время выполнения

(файл меньше, библиотеки общие, но нужна их наличие при запуске).

Загрузка (loading) — размещение готовой программы и её библиотек в оперативной памяти и передача ей управления для выполнения; этим занимается загрузчик ОС.

Общий путь: исходный код → (трансляция) объектный код → (компоновка) исполняемый файл → (загрузка) выполнение в памяти.

11►Задача 11. Библиотеки, компоновщик, связывание, трансляция

Виды библиотек при создании исполняемого файла:

  • Статические библиотеки (.lib, .a) — их код включается в исполняемый файл на

этапе компоновки.

  • Динамические библиотеки (.dll в Windows, .so в Linux) — подключаются во время

выполнения, не входят в сам exe.

Структура создания исполняемого файла:

 исходный код (.c)
        │  компиляция
        ▼
 объектный код (.obj/.o)  +  библиотеки (.lib/.dll)
        │  компоновка (linker)
        ▼
 исполняемый файл (.exe)
        │  загрузка (loader)
        ▼
 выполнение в памяти

Сравнение типов связывания:

  |  Статическое  |  Динамическое
Когда подключается  |  при сборке  |  при выполнении
Размер exe  |  больше  |  меньше
Автономность  |  полная  |  нужна DLL
Обновление библиотеки  |  пересборка  |  замена DLL

Трансляция — пример (Python, интерпретация):

name = input("Имя: ")
print("Привет,", name)

Компиляция — пример (C):

/* greet.c — компилятор создаёт исполняемый файл целиком */
#include <stdio.h>
int main() {
    char name[50];
    printf("Имя: ");
    scanf("%49s", name);
    printf("Привет, %s\n", name);
    return 0;
}
/* Сборка: gcc greet.c -o greet   →   запуск: ./greet */

Компиляция переводит весь код заранее в .exe; интерпретация выполняет код построчно во время работы.

Блок-схема процесса отладки:

        │  Начало │
        └────┬────┘
             ▼
   ┌────────────────────┐
   │ Запуск программы /  │
   │ выполнение теста    │
   └────────┬───────────┘
            ▼
        ◇ Есть ошибка? ◇──── Нет ──► ┌──────────┐
            │ Да                     │  Конец   │
            ▼                        └──────────┘
   ┌────────────────────┐
   │ Локализация ошибки  │ (точки останова,
   │ (поиск места)       │  просмотр переменных)
   └────────┬───────────┘
            ▼
   ┌────────────────────┐
   │ Исправление кода    │
   └────────┬───────────┘
            ▼
   ┌────────────────────┐
   │ Повторный запуск    │
   └────────┬───────────┘
            └──────► (назад к проверке «Есть ошибка?»)

Фрагмент кода с ошибкой и отладкой:

def average(numbers):
    total = 0
    for n in numbers:
        total += n
    return total / len(numbers)   # ошибка: деление на ноль при пустом списке

def average_fixed(numbers):
    if not numbers:
        return 0
    return sum(numbers) / len(numbers)

print(average_fixed([4, 6, 8]))   # 6.0
print(average_fixed([]))          # 0

Работа с инструментом отладки: в IDE ставится точка останова (breakpoint) на нужной строке, программа запускается в режиме отладки, выполнение приостанавливается, и можно посмотреть значения переменных, выполнить код пошагово (Step Over/Into), найти строку, где значение становится некорректным.

Метод тестирования: модульное (unit) тестирование — пример ситуации: проверяем функцию average_fixed на разных входных данных (обычный список, пустой список, один элемент), сравнивая результат с ожидаемым:

assert average_fixed([2, 4]) == 3.0
assert average_fixed([]) == 0

Модульное тестирование применяют, чтобы убедиться в корректной работе отдельной функции в граничных и обычных случаях.
