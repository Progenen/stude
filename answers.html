БЛОК 1. АЛГОРИТМИЗАЦИЯ И ПРОГРАММИРОВАНИЕ

1►Вопрос 1. Алгоритм: понятие, свойства, формы записи, алгоритмические структуры

Понятие алгоритма

Алгоритм — это точная, конечная последовательность однозначно определённых шагов, которая за конечное число шагов приводит к решению поставленной задачи или достижению определённой цели при любых допустимых исходных данных.

Основные свойства алгоритмов

  • Дискретность — алгоритм разбит на отдельные шаги, выполняемые последовательно друг за другом.
  • Детерминированность — каждый шаг однозначно определён: нет двусмысленности в интерпретации команд.
  • Конечность — алгоритм обязательно завершается за конечное число шагов.
  • Результативность — при корректных входных данных алгоритм всегда даёт правильный результат.
  • Массовость — алгоритм применим к целому классу однотипных задач с различными допустимыми исходными данными.
  • Понятность — каждый шаг понятен исполнителю (человеку или компьютеру).

Формы записи алгоритмов

  • Словесная — описание на естественном языке (русском, казахском и т.д.). Наглядна, но допускает неоднозначность.
  • Псевдокод — структурированная запись на условном языке (смесь естественного языка и операторов программирования). Удобна для описания логики до кодирования.
  • Блок-схема — графическое представление: прямоугольник — действие/обработка; ромб — условие/ветвление; параллелограмм — ввод/вывод; овал — начало/конец.
  • Программный код — запись на языке программирования (C++, Python, Java и др.). Непосредственно исполняется машиной.

Основные алгоритмические структуры

  • Линейная (следование) — команды выполняются строго последовательно одна за другой, без ветвлений и повторений.
  • Ветвление (условие) — в зависимости от результата проверки условия выполняется одна из двух ветвей (if-else). Также существует множественное ветвление (switch).
  • Цикл (повторение) — блок инструкций повторяется несколько раз: цикл с параметром (for), с предусловием (while), с постусловием (do-while).
  • Подпрограмма (функция/процедура) — повторяющийся блок оформляется как отдельная именованная единица, вызываемая из разных мест программы.
2►Вопрос 2. Технология программирования. Языки программирования. Программное обеспечение

Технология программирования

Технология программирования — это совокупность методов, инструментов, стандартов и процессов, используемых при создании программного обеспечения. Охватывает полный жизненный цикл ПО: анализ требований → проектирование → кодирование → тестирование → отладку → документирование → сопровождение.

История развития языков программирования

  • 1940-е — машинные коды: программирование в двоичных (или восьмеричных) кодах команд процессора.
  • 1950-е — ассемблер: мнемонические обозначения вместо двоичных кодов (MOV, ADD, JMP).
  • 1957 — FORTRAN (IBM): первый высокоуровневый язык, ориентированный на научные вычисления.
  • 1959 — COBOL: для автоматизации бизнес-приложений и обработки данных.
  • 1960 — ALGOL: ввёл концепцию блочной структуры и структурного программирования.
  • 1972 — C (Деннис Ритчи): системное программирование; на нём написана ОС Unix.
  • 1983 — C++ (Бьёрн Страуструп): объектно-ориентированное расширение языка C.
  • 1991 — Python; 1995 — Java, JavaScript; 2000-е — C#, Swift, Kotlin, Go и другие современные языки.

Программное обеспечение (ПО)

Программное обеспечение — совокупность программ, данных и документации, обеспечивающих функционирование и использование вычислительной техники.

  • Системное ПО — операционные системы (Windows, Linux, macOS), драйверы устройств, утилиты обслуживания системы.
  • Прикладное ПО — офисные пакеты, браузеры, СУБД-клиенты, игры, CAD-системы и т.д.
  • Инструментальное ПО — IDE (Visual Studio, CLion), компиляторы, отладчики, системы контроля версий (Git).
3►Вопрос 3. Интегрированные системы программирования. Трансляторы. Ошибки. Этапы разработки

Состав интегрированных систем программирования (IDE)

  • Редактор исходного кода с подсветкой синтаксиса и автодополнением (IntelliSense).
  • Встроенный компилятор или интерпретатор.
  • Отладчик (debugger) — пошаговое выполнение, точки останова, просмотр переменных.
  • Профилировщик — анализ производительности и использования памяти.
  • Менеджер проектов и система сборки (make, CMake).
  • Примеры IDE: Visual Studio, Code::Blocks, CLion, Eclipse, VS Code, Dev-C++.

Трансляторы и их виды

Транслятор — программа, преобразующая исходный код на языке высокого уровня в машинный код или промежуточное представление.

  • Компилятор — переводит всю программу целиком в исполняемый машинный код до запуска. Результат: .exe или .out файл. Примеры языков: C, C++, Pascal, Delphi.
  • Интерпретатор — выполняет программу построчно, транслируя и исполняя каждый оператор непосредственно. Исполняемый файл не создаётся. Примеры: Python, PHP, JavaScript (в браузере).
  • JIT-компилятор (Just-In-Time) — компилирует программу в машинный код непосредственно во время выполнения, сочетая достоинства обоих подходов. Примеры: Java (JVM), C# (.NET CLR).

Виды ошибок при трансляции программ

  • Синтаксические ошибки — нарушение правил записи конструкций языка (пропущена точка с запятой, несбалансированные скобки). Обнаруживаются компилятором.
  • Семантические ошибки — логически некорректное использование синтаксически правильных конструкций (использование неинициализированной переменной, несоответствие типов).
  • Ошибки времени выполнения (runtime errors) — возникают при исполнении программы: деление на ноль, обращение по нулевому указателю, выход за границы массива.
  • Логические ошибки — программа компилируется и работает без сбоев, но выдаёт неверный результат из-за ошибки в алгоритме.

Этапы разработки программ на ЭВМ

  • 1. Постановка задачи: формулировка цели, определение входных и выходных данных, ограничений.
  • 2. Проектирование алгоритма: построение блок-схемы или псевдокода, выбор структур данных.
  • 3. Написание исходного кода на языке программирования.
  • 4. Компиляция и устранение синтаксических и семантических ошибок.
  • 5. Отладка (debugging): поиск и исправление логических и runtime ошибок.
  • 6. Тестирование: проверка на различных наборах входных данных, включая граничные случаи.
  • 7. Документирование: написание комментариев в коде и технической документации.
  • 8. Сопровождение и обновление (maintenance).
4►Вопрос 4. Основная парадигма C++. Состав языка. Переменные, константы, типы данных, структура программы

Основная парадигма C++

C++ — мультипарадигменный язык программирования. Поддерживает одновременно: процедурное программирование (как в C), объектно-ориентированное программирование (ООП: классы, наследование, полиморфизм), обобщённое программирование (шаблоны — templates) и элементы функционального программирования.

Состав языка C++

  • Алфавит: латинские буквы (a–z, A–Z), цифры (0–9), специальные символы (+, -, *, /, =, {, }, и др.).
  • Ключевые слова (зарезервированные): int, float, double, char, bool, if, else, while, for, do, switch, case, break, return, class, struct, new, delete и др.
  • Идентификаторы: имена переменных, функций, классов. Начинаются с буквы или '_', чувствительны к регистру.
  • Литералы: константные значения в коде — 42 (целое), 3.14 (вещественное), 'A' (символьное), "hello" (строковое), true/false (логическое).
  • Операторы: арифметические, логические, побитовые, операторы сравнения, присваивания и др.
  • Директивы препроцессора: #include (подключение заголовочных файлов), #define (макроопределения), #ifdef / #endif (условная компиляция).

Переменные и константы

Переменная — именованная область памяти, значение которой может изменяться во время выполнения программы. Объявляется с указанием типа.

int age = 20;      // целочисленная переменная
double pi = 3.14;  // вещественная переменная
char grade = 'A';  // символьная переменная

Константа — величина, значение которой установлено один раз и не изменяется в течение работы программы.

const int MAX = 100;     // константа (проверяется компилятором)
#define SIZE 50          // макро-константа (замена препроцессором)

Основные типы данных в C++

  • int — целое число (4 байта): от −2 147 483 648 до 2 147 483 647.
  • float — вещественное число с одинарной точностью (4 байта, ≈7 значащих цифр).
  • double — вещественное число с двойной точностью (8 байт, ≈15 значащих цифр).
  • char — символ в кодировке ASCII (1 байт).
  • bool — логический тип: true (1) или false (0).
  • long long — расширенное целое (8 байт); unsigned — беззнаковое целое; short — короткое целое (2 байта).

Структура программы в C++

#include <iostream>      // подключение стандартной библиотеки ввода-вывода
using namespace std;    // использование пространства имён std
int main() {            // точка входа в программу
int a = 5;          // объявление и инициализация переменной
cout << a << endl;  // вывод значения на экран
return 0;           // код завершения 0 = успех
}
5►Вопрос 5. Операции C++: арифметические операции, математические функции, выражения

Арифметические операции

  • + — сложение: 5 + 3 = 8
  • − — вычитание: 10 − 4 = 6
  • * — умножение: 6 * 7 = 42
  • / — деление: для int — целочисленное (7 / 2 = 3); для double — вещественное (7.0 / 2 = 3.5).
  • % — остаток от деления (только для целых): 10 % 3 = 1.

💡 Приоритет: * / % выполняются раньше + −. Скобки меняют порядок: (2 + 3) * 4 = 20.

Стандартные математические функции (библиотека <cmath>)

  • sqrt(x) — квадратный корень из x (x ≥ 0).
  • pow(x, y) — x в степени y.
  • fabs(x) / abs(x) — модуль (абсолютное значение) числа.
  • sin(x), cos(x), tan(x) — тригонометрические функции (аргумент в радианах).
  • asin(x), acos(x), atan(x) — обратные тригонометрические функции.
  • log(x) — натуральный логарифм ln(x); log10(x) — десятичный логарифм.
  • ceil(x) — округление вверх; floor(x) — округление вниз; round(x) — обычное округление.

Примеры арифметических выражений

#include <iostream>
#include <cmath>
using namespace std;
int main() {
int a = 10, b = 3;
cout << a + b;           // 13
cout << a / b;           // 3  (целочисленное деление!)
cout << (double)a / b;   // 3.333...  (приведение типа)
cout << a % b;           // 1
// Гипотенуза прямоугольного треугольника:
double hyp = sqrt(pow(3.0, 2) + pow(4.0, 2)); // 5.0
cout << hyp;
return 0;
}
6►Вопрос 6. Логические операции C++. Логические выражения

Логические операции

  • && — логическое И (AND): результат true только если оба операнда true.
  • || — логическое ИЛИ (OR): результат true если хотя бы один операнд true.
  • ! — логическое НЕ (NOT): инвертирует значение: !true = false, !false = true.

Операции сравнения (возвращают bool)

  • == — равно; != — не равно.
  • < , > — строго меньше / строго больше.
  • <= , >= — меньше или равно / больше или равно.

💡 Частая ошибка: использование = (присваивание) вместо == (сравнение) в условии.

Приоритет логических операций

! (наивысший) → && → || (наименьший). Скобки помогают сделать код читаемым.

Примеры логических выражений

int a = 5, b = 10, c = 0;
bool r1 = (a > 0) && (b < 20);      // true && true = true
bool r2 = (a == 5) || (b == 5);     // true || false = true
bool r3 = !(a > 3);                 // !(true) = false
bool r4 = (a != b) && (a < b);      // true && true = true
bool r5 = (c != 0) && (a / c > 0); // false && ... = false (c=0 не вычисляется — короткое замыкание!)

💡 Короткое замыкание (short-circuit evaluation): в выражении A && B, если A = false, B не вычисляется. В A || B, если A = true, B не вычисляется.

Механизм передачи аргументов

  • Передача по значению (по умолчанию) — функция получает копию аргумента. Изменения внутри функции не влияют на оригинал.

15►Вопрос 15. Рекурсия в C++. Примеры рекурсивных алгоритмов

Определение рекурсии

Рекурсия — это способ определения функции, при котором функция вызывает саму себя. Каждый рекурсивный алгоритм обязательно содержит:

  • Базовый случай (условие выхода) — условие, при котором рекурсия прекращается и функция возвращает результат без рекурсивного вызова.
  • Рекурсивный случай — вызов функции с изменёнными аргументами, которые приближают задачу к базовому случаю.
// Факториал: n! = n * (n-1) * ... * 1,  0! = 1
long long factorial(int n) {
if (n <= 0) return 1;            // базовый случай
return n * factorial(n - 1);     // рекурсивный вызов
}
cout << factorial(5);  // 120
// Числа Фибоначчи: F(n) = F(n-1) + F(n-2), F(0)=0, F(1)=1
int fib(int n) {
if (n <= 1) return n;            // базовый случай
return fib(n - 1) + fib(n - 2); // рекурсивный вызов
}
cout << fib(10);  // 55

💡 Рекурсия использует стек вызовов. При слишком глубокой рекурсии возможно переполнение стека (stack overflow). Для чисел Фибоначчи рекурсия экспоненциально медленная — лучше использовать итерацию или мемоизацию.

20►Вопрос 20. Сортировка. Методы внутренней сортировки

Определение сортировки

Сортировка — упорядочение элементов массива (или другой структуры данных) по некоторому критерию (по возрастанию, убыванию, алфавиту и т.д.).

Основные методы внутренней сортировки (данные в памяти)

  • Сортировка выбором (Selection Sort) — на каждом шаге находим минимальный элемент неотсортированной части и ставим его в начало. Сложность: O(n²).
  • Пузырьковая сортировка (Bubble Sort) — многократно сравниваем соседние элементы и меняем местами при нарушении порядка. Сложность: O(n²), в лучшем случае O(n) с оптимизацией.
  • Сортировка вставками (Insertion Sort) — каждый новый элемент вставляется на нужную позицию в уже отсортированную часть. Эффективна для почти отсортированных данных. Сложность: O(n²).
  • Быстрая сортировка (Quick Sort) — выбирается опорный элемент (pivot), массив делится на две части, каждая рекурсивно сортируется. Средняя сложность: O(n log n).
  • Сортировка слиянием (Merge Sort) — массив делится пополам, обе части сортируются рекурсивно, затем сливаются. Сложность: O(n log n), устойчивая.
21►Вопрос 21. Сортировка выборкой (выбором)

Стратегия сортировки простой выборкой

На каждой итерации i находим индекс минимального элемента в неотсортированной части [i..n-1] и меняем его с элементом arr[i]. После i-й итерации первые i+1 элементов стоят на правильных местах.

void selectionSort(int arr[], int n) {
for (int i = 0; i < n - 1; i++) {
int minIdx = i;                       // предполагаем, что i-й минимальный
for (int j = i + 1; j < n; j++)
if (arr[j] < arr[minIdx])
minIdx = j;                  // нашли элемент меньше
if (minIdx != i)
swap(arr[i], arr[minIdx]);        // ставим минимум на место
}
}
// Пример: {5, 1, 3, 2, 4}
// i=0: min=arr[1]=1, swap → {1, 5, 3, 2, 4}
// i=1: min=arr[3]=2, swap → {1, 2, 3, 5, 4}
// i=2: min=arr[2]=3, нет swap
// i=3: min=arr[4]=4, swap → {1, 2, 3, 4, 5}

💡 Количество сравнений: n(n-1)/2. Количество обменов: O(n) — меньше, чем у пузырьковой. Не зависит от исходного порядка элементов.

22►Вопрос 22. Пузырьковая сортировка (Bubble Sort)

Стратегия обменной сортировки

Многократно проходим по массиву. На каждом проходе сравниваем каждую соседнюю пару элементов и меняем их местами, если они стоят в неправильном порядке. Крупные элементы «всплывают» (как пузырьки) в конец массива.

// Базовая реализация
void bubbleSort(int arr[], int n) {
for (int i = 0; i < n - 1; i++)
for (int j = 0; j < n - 1 - i; j++)
if (arr[j] > arr[j + 1])
swap(arr[j], arr[j + 1]);
}
// Оптимизированная версия (с флагом остановки)
void bubbleSortOpt(int arr[], int n) {
for (int i = 0; i < n - 1; i++) {
bool swapped = false;
for (int j = 0; j < n - 1 - i; j++)
if (arr[j] > arr[j + 1]) {
swap(arr[j], arr[j + 1]);
swapped = true;
}
if (!swapped) break;  // массив уже отсортирован
}
}
23►Вопрос 23. Алгоритмы поиска. Линейный поиск

Линейный (последовательный) поиск

Поочерёдно просматриваем все элементы массива, пока не найдём искомый элемент или не дойдём до конца. Работает на любом массиве (отсортированном и нет).

// Возвращает индекс найденного элемента, или -1 если не найден
int linearSearch(int arr[], int n, int key) {
for (int i = 0; i < n; i++)
if (arr[i] == key) return i;
return -1;
}
int arr[] = {4, 7, 1, 9, 3};
int idx = linearSearch(arr, 5, 9);
if (idx != -1) cout << "Найдено на позиции: " << idx;  // 3
else cout << "Не найдено";

💡 Временная сложность: O(n) в худшем случае (элемент в конце или отсутствует). Преимущество: простота реализации, работает на любом массиве.

24►Вопрос 24. Двоичный (бинарный) поиск

Принцип работы

Применяется только к отсортированному массиву. Алгоритм: сравниваем искомый элемент со средним элементом массива; если равен — найдено; если меньше — ищем в левой половине; если больше — в правой. На каждом шаге диапазон поиска уменьшается вдвое.

int binarySearch(int arr[], int n, int key) {
int left = 0, right = n - 1;
while (left <= right) {
int mid = left + (right - left) / 2;  // безопасный расчёт (нет переполнения)
if (arr[mid] == key)  return mid;     // найдено!
else if (arr[mid] < key) left  = mid + 1;  // ищем правее
else                     right = mid - 1;  // ищем левее
}
return -1;  // не найдено
}
// Пример: arr = {1, 3, 5, 7, 9, 11}, ищем 7
// mid=2 (arr[2]=5 < 7) → left=3
// mid=4 (arr[4]=9 > 7) → right=3
// mid=3 (arr[3]=7 == 7) → найдено! индекс 3

💡 Временная сложность: O(log n). При n = 1 000 000 потребуется не более 20 сравнений! Обязательное условие: массив отсортирован.

if (!head) return;
if (head->data == val) {
Node *tmp = head;
head = head->next;
delete tmp; return;
}
Node *curr = head;
while (curr->next && curr->next->data != val)
curr = curr->next;
if (curr->next) {

curr->next = tmp->next;
delete tmp;
}
}
30►Вопрос 30. Динамические структуры данных: деревья

Определение и терминология

Дерево — нелинейная иерархическая структура, где каждый узел имеет ровно одного родителя (кроме корня) и произвольное количество дочерних узлов (потомков).

  • Корень (root) — единственный узел без родителя.
  • Листья (leaves) — узлы без дочерних узлов.
  • Высота дерева — длина максимального пути от корня до листа.

Бинарное дерево поиска (BST)

Свойство: для каждого узла все ключи в левом поддереве меньше ключа узла, все ключи в правом поддереве больше.

struct TreeNode {
int data;
TreeNode *left, *right;
};
// Вставка
TreeNode* insert(TreeNode *root, int val) {
if (!root) return new TreeNode{val, nullptr, nullptr};
if (val < root->data) root->left  = insert(root->left,  val);
else                  root->right = insert(root->right, val);
return root;
}
// Обход Inorder (LNR) — выдаёт элементы в отсортированном порядке
void inorder(TreeNode *root) {
if (!root) return;
inorder(root->left);
cout << root->data << " ";
inorder(root->right);
}
// Поиск
bool search(TreeNode *root, int key) {
if (!root) return false;
if (root->data == key) return true;
if (key < root->data) return search(root->left, key);
return search(root->right, key);
}

💡 Обходы: Preorder (NLR): корень → лево → право; Inorder (LNR): лево → корень → право; Postorder (LRN): лево → право → корень.

1►Вопрос 1. База данных и СУБД: определения и отличия

База данных (БД)

База данных — организованная совокупность структурированных данных, хранящихся в электронном виде, связанных единой логикой и отражающих определённую предметную область.

СУБД (Система управления базами данных)

СУБД — программный комплекс для создания, хранения, обновления, защиты и управления базами данных. Обеспечивает интерфейс между пользователем/приложением и базой данных.

Примеры: Microsoft SQL Server, MySQL, PostgreSQL, Oracle Database, SQLite, MariaDB.

Ключевое различие

  • БД — это сами данные (таблицы, записи, файлы хранения).
  • СУБД — это программа для работы с данными: создание объектов, выполнение запросов, управление доступом, обеспечение целостности и безопасности данных.
  • Аналогия: БД — это библиотечный фонд (книги), СУБД — это библиотечная система (каталог, правила выдачи, читальный зал).
2►Вопрос 2. Архитектура СУБД. Модели архитектур
  • Файл-серверная архитектура — сервер хранит файлы данных, вся обработка выполняется на клиенте. Низкая производительность при многопользовательском доступе из-за большого трафика. Пример: ранние версии Access.
  • Клиент-серверная архитектура — сервер СУБД принимает SQL-запросы от клиентов и возвращает только результаты. Значительно эффективнее. Примеры: MySQL, SQL Server, PostgreSQL.
  • Трёхзвенная (N-tier) архитектура — клиент ↔ сервер приложений (бизнес-логика) ↔ сервер БД. Высокая масштабируемость, безопасность, используется в веб-приложениях.
  • Облачная архитектура — СУБД и данные размещены в облачной инфраструктуре. Высокая доступность, масштабируемость, оплата по потреблению. Примеры: Amazon RDS, Azure SQL, Google Cloud SQL.
3►Вопрос 3. Трёхуровневая архитектура БД (ANSI/SPARC)

Модель ANSI/SPARC (1975) определяет три уровня абстракции для описания базы данных, обеспечивающих независимость данных.

Три уровня

  • Внешний уровень (уровень представлений) — индивидуальный «вид» данных для конкретного пользователя или приложения. Реализуется через представления (VIEW). Разные пользователи видят разные подмножества данных.
  • Концептуальный уровень (логический) — полное описание структуры всей базы данных без деталей физического хранения: таблицы, атрибуты, связи, ограничения целостности.
  • Внутренний уровень (физический) — описание физического способа хранения данных на носителе: файлы, страницы, индексы, методы доступа, хэш-таблицы.

Преимущество

Физическая независимость данных: изменение физического хранилища (например, добавление индекса) не меняет концептуальный уровень. Логическая независимость: изменение логической схемы не ломает приложения, работающие с внешним уровнем.

4►Вопрос 4. Модели данных. Классификация

Что такое модель данных

Модель данных — формальное описание структуры данных, операций над ними и ограничений целостности. Определяет способ организации, хранения и доступа к данным.

Классификация моделей данных

  • Иерархическая модель — данные организованы в виде дерева (один родитель). Быстрый доступ по иерархии, но сложно представлять отношения M:N. Пример: IBM IMS.
  • Сетевая модель — обобщение иерархической: запись может иметь несколько родителей. Гибкость, но сложная структура. Пример: IDMS.
  • Реляционная модель — данные в двумерных таблицах (отношениях), связи через ключи. Математическая основа — реляционная алгебра. Язык запросов — SQL. Наиболее распространена.
  • Объектно-ориентированная модель — данные как объекты с атрибутами и методами, поддержка наследования. Примеры: ObjectDB.
  • Документоориентированная модель — хранение документов в формате JSON/BSON. Примеры: MongoDB, CouchDB.
  • Ключ-значение (Key-Value) — простейшая модель: пары «ключ — значение». Примеры: Redis, DynamoDB.
  • Колоночная модель — данные хранятся по столбцам, эффективна для аналитики. Примеры: Apache Cassandra, ClickHouse.
5►Вопрос 5. Реляционная алгебра: объединение и пересечение

Объединение (Union — ∪)

Возвращает все кортежи, входящие хотя бы в одно из двух отношений (дубликаты исключаются). Условие совместимости: оба отношения должны иметь одинаковое число атрибутов с совместимыми доменами.

Синтаксис в реляционной алгебре: R ∪ S

-- SQL

SELECT * FROM R

UNION

SELECT * FROM S;

-- UNION ALL сохраняет дубликаты

Пересечение (Intersection — ∩)

Возвращает только те кортежи, которые присутствуют в обоих отношениях одновременно.

Синтаксис в реляционной алгебре: R ∩ S

-- SQL

SELECT * FROM R

INTERSECT

SELECT * FROM S;
6►Вопрос 6. Реляционная алгебра: разность и декартово произведение

Разность (Set Difference — \)

Возвращает кортежи, которые есть в R, но отсутствуют в S.

Синтаксис: R \ S (или R − S)

-- SQL (MS SQL Server)

SELECT * FROM R

EXCEPT

SELECT * FROM S;

-- В Oracle: MINUS вместо EXCEPT

Декартово произведение (Cartesian Product — ×)

Возвращает все возможные комбинации кортежей из R и S. Если R содержит m строк, S — n строк, результат содержит m × n строк. Атрибуты: объединение атрибутов R и S.

Синтаксис: R × S

-- SQL

SELECT * FROM R CROSS JOIN S;

-- Пример: R(3 строки) × S(4 строки) → 12 строк в результате

-- Используется как основа для операции соединения (JOIN)

7►Вопрос 7. Основные концепции реляционных БД. Соответствие терминов

Соответствие понятий реляционной модели и таблицы

  • Отношение = Таблица — двумерная структура с именованными столбцами.
  • Кортеж = Строка (запись) — одна запись в таблице.
  • Атрибут = Столбец (поле) — характеристика объекта.
  • Домен = Множество допустимых значений атрибута. Например, домен «Возраст» — целые числа от 0 до 150.
  • Степень отношения = Количество атрибутов (столбцов) в таблице.
  • Мощность отношения = Количество кортежей (строк) в таблице.
  • Сущность = Объект реального мира, о котором хранятся данные (Студент, Товар, Сотрудник).
  • Схема отношения = Структура таблицы: имя + перечень атрибутов с их доменами.
8►Вопрос 8. Ключи в реляционных БД

Определение ключа

Ключ — атрибут или минимальный набор атрибутов, значения которых однозначно идентифицируют каждый кортеж в отношении.

Виды ключей

  • Суперключ — любой набор атрибутов, однозначно идентифицирующий строку (может содержать лишние атрибуты).
  • Потенциальный ключ — минимальный суперключ (нельзя удалить ни один атрибут, не потеряв свойство уникальности). В таблице может быть несколько.
  • Первичный ключ (PRIMARY KEY) — один из потенциальных ключей, выбранный как основной. Значение обязательно (NOT NULL) и уникально. Одна таблица — один первичный ключ.
  • Уникальный ключ (UNIQUE) — альтернативный потенциальный ключ. Допускает NULL. Таблица может иметь несколько UNIQUE-ограничений.
  • Внешний ключ (FOREIGN KEY) — атрибут(ы) одной таблицы, ссылающийся(еся) на первичный ключ другой таблицы. Обеспечивает ссылочную целостность.
  • Составной ключ — ключ, состоящий из двух и более атрибутов. Применяется, когда один атрибут не обеспечивает уникальности.
CREATE TABLE Orders (

OrderID INT PRIMARY KEY, -- первичный ключ

Email VARCHAR(100) UNIQUE, -- уникальный ключ

CustomerID INT FOREIGN KEY REFERENCES Customers(CustomerID), -- внешний ключ

OrderDate DATE

);
9►Вопрос 9. Ограничения реляционной модели на таблицы БД
  • Атомарность значений (1НФ) — каждая ячейка содержит единственное, неделимое значение. Нельзя хранить списки или составные значения в одном поле.
  • Уникальность строк — не может быть двух одинаковых кортежей в одном отношении.
  • Ограничение первичного ключа — PRIMARY KEY: значение уникально, не может быть NULL, одно на таблицу.
  • Ссылочная целостность — FOREIGN KEY: значение внешнего ключа должно соответствовать существующему значению первичного ключа в родительской таблице (или быть NULL).
  • Доменные ограничения — значения атрибутов должны принадлежать заданному домену (тип данных, диапазон).
  • Ограничение NOT NULL — запрет на NULL-значения в определённых столбцах.
  • Проверочное ограничение CHECK — значения должны удовлетворять указанному условию: CHECK (age BETWEEN 0 AND 150).
  • Ограничение DEFAULT — значение по умолчанию, если при вставке поле не указано.
10►Вопрос 10. Нормализация базы данных. Нормальные формы

Суть нормализации

Нормализация — процесс приведения структуры БД к виду, исключающему избыточность данных и аномалии при вставке (INSERT), обновлении (UPDATE) и удалении (DELETE) данных.

Нормальные формы

  • 1НФ — все атрибуты атомарны (неделимы), нет повторяющихся групп, каждая строка уникальна (есть первичный ключ).
  • 2НФ — выполнена 1НФ + каждый неключевой атрибут полностью функционально зависит от всего первичного ключа (нет частичных зависимостей). Актуально для составных ключей.
  • 3НФ — выполнена 2НФ + нет транзитивных зависимостей: неключевые атрибуты не зависят от других неключевых атрибутов (только от первичного ключа).
  • НФБК (Бойса-Кодда, BCNF) — усиленная 3НФ: каждый детерминант является потенциальным ключом.
  • 4НФ — выполнена НФБК + нет нетривиальных многозначных зависимостей.
  • 5НФ — выполнена 4НФ + нет нетривиальных зависимостей соединения.

💡 На практике обычно нормализуют до 3НФ или НФБК. Дальнейшая нормализация может снизить производительность запросов.

11►Вопрос 11. ER-диаграммы. Основные понятия

Назначение ER-диаграмм

ER-диаграмма (Entity-Relationship, сущность-связь) — графическое средство концептуального моделирования данных, предложенное Питером Ченом в 1976 году. Используется для визуализации предметной области до создания таблиц в СУБД.

Основные элементы

  • Сущность (Entity) — объект реального мира, о котором хранятся данные (Студент, Книга, Товар). Изображается прямоугольником.
  • Атрибут — характеристика сущности (ФИО, год рождения, цена). Изображается эллипсом. Ключевой атрибут — подчёркивается.
  • Связь (Relationship) — ассоциация между сущностями (Студент УЧИТСЯ в Группе). Изображается ромбом с линиями к сущностям.
  • Кардинальность связи — 1:1 (один к одному), 1:N (один ко многим), M:N (многие ко многим).
  • Слабая сущность — сущность, не имеющая собственного ключа и идентифицируемая только вместе с родительской сущностью.
12►Вопрос 12. Виды связей между таблицами
  • Один к одному (1:1) — одной записи в таблице A соответствует максимум одна запись в таблице B, и наоборот. Пример: Сотрудник ↔ Паспорт. Реализация: внешний ключ в любой из таблиц с ограничением UNIQUE.
  • Один ко многим (1:N) — одной записи в A соответствует несколько записей в B, а каждая запись B связана с одной записью A. Наиболее распространённый тип. Пример: Отдел → Сотрудники. Реализация: внешний ключ в таблице «многие» (B).
  • Многие ко многим (M:N) — многим записям A соответствуют многие записи B. Пример: Студент ↔ Дисциплина (студент изучает много дисциплин, дисциплину изучают много студентов). Реализация: промежуточная (ассоциативная) таблица с двумя внешними ключами.

-- Реализация связи M:N: Студент — Дисциплина

CREATE TABLE StudentDiscipline (

StudentID INT FOREIGN KEY REFERENCES Student(StudentID),

DisciplineID INT FOREIGN KEY REFERENCES Discipline(DisciplineID),

Grade INT,

PRIMARY KEY (StudentID, DisciplineID) -- составной ключ

);
13►Вопрос 13. Язык SQL. Структура. Функции

SQL (Structured Query Language)

SQL — декларативный стандартный язык для работы с реляционными базами данных, принятый как стандарт ISO/ANSI. Описывает что нужно получить, а не как. Нечувствителен к регистру.

Структура SQL (категории операторов)

  • DDL (Data Definition Language) — операторы определения данных: CREATE (создание объектов), ALTER (изменение структуры), DROP (удаление объектов), TRUNCATE (очистка таблицы).
  • DML (Data Manipulation Language) — операторы манипулирования данными: SELECT (выборка), INSERT (вставка), UPDATE (обновление), DELETE (удаление).
  • DCL (Data Control Language) — операторы управления доступом: GRANT (предоставить права), REVOKE (отозвать права).
  • TCL (Transaction Control Language) — операторы управления транзакциями: BEGIN TRANSACTION, COMMIT (зафиксировать), ROLLBACK (откатить), SAVEPOINT (точка сохранения).
14►Вопрос 14. DDL: оператор CREATE TABLE

Создаёт новую таблицу в базе данных с определением столбцов, типов данных и ограничений.

CREATE TABLE Student (

StudentID INT PRIMARY KEY IDENTITY(1,1), -- автоинкремент

IIN CHAR(12) NOT NULL UNIQUE, -- ИИН уникален

FullName NVARCHAR(100) NOT NULL,

BirthYear INT CHECK (BirthYear BETWEEN 1900 AND 2010),

Course INT DEFAULT 1,

Program NVARCHAR(100),

GroupID INT FOREIGN KEY REFERENCES Groups(GroupID)

);
15►Вопрос 15. DDL: оператор DROP
DROP безвозвратно удаляет объект базы данных вместе со всеми данными. Операция необратима без резервной копии.
DROP TABLE Student;              -- удаление таблицы
DROP VIEW StudentView;           -- удаление представления
DROP INDEX idx_name ON Student;  -- удаление индекса
DROP PROCEDURE GetStudents;      -- удаление хранимой процедуры
DROP DATABASE TestDB;            -- удаление всей базы данных!

-- Безопасное удаление (не выдаёт ошибку если объект не существует):

DROP TABLE IF EXISTS Student;

💡 При наличии внешних ключей, ссылающихся на таблицу, сначала нужно удалить зависимые таблицы или ограничения. Отличие от TRUNCATE: DROP удаляет структуру таблицы, TRUNCATE только очищает данные.

16►Вопрос 16. DDL: оператор ALTER TABLE

Изменяет структуру существующей таблицы: добавляет/удаляет столбцы, изменяет типы данных, управляет ограничениями.

-- Добавить столбец

ALTER TABLE Student ADD Address NVARCHAR(200);
ALTER TABLE Student ADD Email NVARCHAR(100) DEFAULT 'unknown@mail.kz';

-- Изменить тип/размер столбца

ALTER TABLE Student ALTER COLUMN FullName NVARCHAR(200);

-- Удалить столбец

ALTER TABLE Student DROP COLUMN Address;

-- Добавить ограничение

ALTER TABLE Student ADD CONSTRAINT chk_year
CHECK (BirthYear > 1900 AND BirthYear < 2015);

-- Добавить внешний ключ

ALTER TABLE Student ADD CONSTRAINT fk_group
FOREIGN KEY (GroupID) REFERENCES Groups(GroupID);

-- Удалить ограничение

ALTER TABLE Student DROP CONSTRAINT chk_year;
17►Вопрос 17. DDL: CREATE VIEW (представление)

Представление (VIEW) — виртуальная таблица, основанная на результате запроса SELECT. Данные не хранятся физически — при обращении к представлению выполняется базовый запрос.

-- Создание представления

CREATE VIEW ActiveStudents AS
SELECT s.StudentID, s.FullName, s.BirthYear, g.GroupName

FROM Student s

JOIN Groups g ON s.GroupID = g.GroupID

WHERE s.IsActive = 1;

-- Использование как обычной таблицы

SELECT * FROM ActiveStudents WHERE GroupName = 'ИС-21';

-- Изменение или удаление

ALTER VIEW ActiveStudents AS SELECT ...;
DROP VIEW ActiveStudents;

Назначение представлений

  • Упрощение сложных запросов — сложный JOIN записывается один раз.
  • Контроль доступа — пользователь видит только разрешённые столбцы/строки.
  • Логическая независимость данных — приложение обращается к представлению, а не к таблице.
18►Вопрос 18. DML: DELETE — удаление данных

-- Удалить конкретную запись

DELETE FROM Student WHERE StudentID = 101;

-- Удалить по нескольким условиям

DELETE FROM Student WHERE BirthYear < 1990 AND Course > 4;

-- Удалить записи с подзапросом

DELETE FROM Orders WHERE CustomerID IN (
SELECT CustomerID FROM Customers WHERE City = 'Алматы'

-- или быстрее: TRUNCATE TABLE Student; (но без WHERE и не в транзакции в некоторых СУБД)

💡 DELETE записывается в журнал транзакций и может быть отменён через ROLLBACK. TRUNCATE нельзя отменить в большинстве СУБД.

21►Вопрос 21. Оператор SELECT. Поиск шаблонов. Встроенные функции

-- Полный синтаксис

SELECT [DISTINCT] выражение_столбцов

FROM таблица

[WHERE условие_строк]

[GROUP BY столбцы_группировки]

[HAVING условие_групп]

[ORDER BY столбцы [ASC|DESC]];

-- Порядок выполнения: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY

Поиск шаблонов (LIKE)

WHERE FullName LIKE 'А%' -- начинается на 'А'

WHERE FullName LIKE '%ов' -- заканчивается на 'ов'

WHERE FullName LIKE '%ан%' -- содержит 'ан' где угодно

WHERE Phone LIKE '+7___7%' -- маска: _ = один любой символ

-- Для поиска самого % или _ используйте ESCAPE:

WHERE Name LIKE '50\%' ESCAPE '\' -- ищет '50%'

Агрегатные функции

  • COUNT(*) — количество строк; COUNT(col) — количество не-NULL значений.
  • SUM(col) — сумма числовых значений столбца.
  • AVG(col) — среднее значение (не-NULL).
  • MAX(col) / MIN(col) — максимум / минимум.
SELECT Department,

COUNT(*) AS TotalEmployees,

AVG(Salary) AS AvgSalary,

MAX(Salary) AS MaxSalary

FROM Employees

GROUP BY Department

HAVING AVG(Salary) > 100000

ORDER BY AvgSalary DESC;
22►Вопрос 22. Связь таблиц. Виды соединений JOIN
  • INNER JOIN — возвращает только строки, имеющие соответствие в обеих таблицах. Самый распространённый тип.
  • LEFT [OUTER] JOIN — все строки из левой таблицы + совпадающие строки из правой. Если совпадения нет — NULL в столбцах правой.
  • RIGHT [OUTER] JOIN — все строки из правой + совпадающие из левой. Можно заменить LEFT JOIN с переставленными таблицами.
  • FULL [OUTER] JOIN — все строки из обеих таблиц. Несовпадающие строки дополняются NULL.
  • CROSS JOIN — декартово произведение: все комбинации строк.
  • SELF JOIN — соединение таблицы с самой собой (с использованием псевдонимов).

-- INNER JOIN: студенты и их группы

SELECT s.FullName, g.GroupName, s.BirthYear

FROM Student s

INNER JOIN Groups g ON s.GroupID = g.GroupID;

-- LEFT JOIN: все группы, даже без студентов

SELECT g.GroupName, COUNT(s.StudentID) AS StudentCount

FROM Groups g

LEFT JOIN Student s ON g.GroupID = s.GroupID

GROUP BY g.GroupName;

-- SELF JOIN: найти сотрудников и их руководителей

SELECT e.FullName AS Employee, m.FullName AS Manager

FROM Employees e

LEFT JOIN Employees m ON e.ManagerID = m.EmployeeID;
23►Вопрос 23. Хранимые процедуры

Хранимая процедура — именованный, предварительно скомпилированный и сохранённый в базе данных блок SQL-кода с возможностью входных и выходных параметров.

Преимущества

  • Производительность: компилируется один раз, хранится в БД.
  • Повторное использование: вызывается из любого приложения.
  • Безопасность: пользователь вызывает процедуру без прямого доступа к таблицам.
  • Меньший сетевой трафик: передаётся только вызов, а не весь SQL-код.

-- Создание процедуры

CREATE PROCEDURE GetStudentsByGroup

@GroupID INT, -- входной параметр

@MinCourse INT = 1 -- с дефолтным значением

AS

BEGIN
SELECT StudentID, FullName, Course

FROM Student

WHERE GroupID = @GroupID AND Course >= @MinCourse

ORDER BY FullName;
END;

-- Вызов

EXEC GetStudentsByGroup @GroupID = 3;
EXEC GetStudentsByGroup @GroupID = 3, @MinCourse = 2;

-- Удаление

DROP PROCEDURE GetStudentsByGroup;
24►Вопрос 24. Пользовательские функции в SQL

Скалярная функция (возвращает одно значение)

CREATE FUNCTION dbo.GetAge(@BirthYear INT)

RETURNS INT

AS

BEGIN
RETURN YEAR(GETDATE()) - @BirthYear;
END;

-- Использование в запросе

SELECT FullName, dbo.GetAge(BirthYear) AS Age

FROM Student

WHERE dbo.GetAge(BirthYear) BETWEEN 18 AND 25;

Табличная функция (возвращает набор строк)

CREATE FUNCTION dbo.GetStudentsByGroup(@GroupID INT)

RETURNS TABLE

AS

RETURN (

SELECT StudentID, FullName, BirthYear, Course

FROM Student

WHERE GroupID = @GroupID

);

-- Использование в FROM

SELECT * FROM dbo.GetStudentsByGroup(5);

Сравнение функций и хранимых процедур

  • Функция обязательно возвращает значение; процедура может возвращать данные через OUTPUT-параметры.
  • Функцию можно использовать в SELECT, WHERE, JOIN; процедуру — только через EXEC/EXECUTE.
  • Функция не может модифицировать данные в таблицах (только чтение); процедура может.
25►Вопрос 25. Триггеры

Триггер — специальная хранимая процедура, автоматически выполняемая СУБД при наступлении указанного события (INSERT, UPDATE, DELETE) в таблице или представлении.

Типы триггеров (MS SQL Server)

  • AFTER (FOR) — выполняется после успешного выполнения операции. Наиболее распространён.
  • INSTEAD OF — выполняется вместо операции. Используется для представлений.

Специальные таблицы внутри триггера

  • INSERTED — содержит новые строки (при INSERT и UPDATE).
  • DELETED — содержит старые строки (при DELETE и UPDATE).

-- Пример: журналирование вставок

CREATE TRIGGER trg_StudentInsert

ON Student

AFTER INSERT

AS

BEGIN
INSERT INTO AuditLog (TableName, Action, RecordID, ActionDate)
SELECT 'Student', 'INSERT', StudentID, GETDATE()
FROM INSERTED;
END;

-- Пример: запрет удаления активных сотрудников

CREATE TRIGGER trg_PreventDelete

ON Employees

INSTEAD OF DELETE

AS

BEGIN

IF EXISTS (SELECT * FROM DELETED WHERE IsActive = 1)

RAISERROR ('Нельзя удалить активного сотрудника!', 16, 1);

ELSE

DELETE FROM Employees WHERE EmployeeID IN
(SELECT EmployeeID FROM DELETED);
END;
26►Вопрос 26. Транзакции. Целостность БД. Точки сохранения

Транзакция

Транзакция — логически неделимая последовательность операций с базой данных, которая выполняется как единое целое: либо все операции успешно завершаются (COMMIT), либо ни одна из них не применяется (ROLLBACK).

Свойства ACID

  • A — Atomicity (Атомарность): транзакция выполняется полностью или не выполняется вообще.
  • C — Consistency (Согласованность): транзакция переводит БД из одного корректного состояния в другое корректное.
  • I — Isolation (Изолированность): параллельные транзакции не мешают друг другу.
  • D — Durability (Долговечность): зафиксированные изменения сохраняются даже при аппаратном сбое.

-- Пример: перевод денег между счетами

BEGIN TRANSACTION;
UPDATE Account SET Balance = Balance - 5000 WHERE AccountID = 1;
UPDATE Account SET Balance = Balance + 5000 WHERE AccountID = 2;

IF @@ERROR = 0

COMMIT;       -- зафиксировать изменения

ELSE

ROLLBACK;     -- откатить все изменения

Точки сохранения (SAVEPOINT)

Позволяют выполнить частичный откат внутри транзакции — не до начала, а до заданной точки.

BEGIN TRANSACTION;
INSERT INTO Orders VALUES (...);

SAVE TRANSACTION sp_after_order; -- точка сохранения

INSERT INTO OrderItems VALUES (...);

IF @@ERROR != 0

ROLLBACK TRANSACTION sp_after_order;  -- откат только до точки
COMMIT;
27►Вопрос 27. Методы защиты информации в БД

Защита от несанкционированного доступа

  • Аутентификация — проверка подлинности пользователя: пароли, токены, двухфакторная аутентификация (2FA), биометрия.
  • Авторизация — управление правами доступа через SQL: GRANT (предоставить), REVOKE (отозвать), DENY (запретить). Принцип минимальных привилегий.
  • Шифрование — защита данных при хранении (TDE — Transparent Data Encryption) и при передаче (TLS/SSL).
  • Аудит и журналирование — запись всех действий пользователей в журнале (кто, что, когда).
  • Роли и группы пользователей — группировка прав: db_reader, db_writer, db_admin и т.д.
  • Маскирование данных — скрытие конфиденциальных данных (Dynamic Data Masking).

Защита от ошибок пользователя

  • Транзакции с ROLLBACK — возможность отменить ошибочно внесённые изменения.
  • Ограничения целостности — CHECK, NOT NULL, FK автоматически предотвращают некорректные данные.
  • Триггеры — автоматическая проверка бизнес-правил при изменении данных.
  • Резервное копирование (Backup) — полные, дифференциальные и инкрементальные резервные копии.
  • Репликация — копирование данных на резервный сервер для восстановления при сбоях.
28►Вопрос 28. Анализ предметной области и функциональность приложения БД

Анализ предметной области — изучение и формализация объектов, процессов, связей и бизнес-правил той сферы деятельности, для которой создаётся база данных (например, библиотека, банк, учебное заведение).

Влияние анализа на функциональность

  • Полнота сущностей — выявление всех значимых объектов: какие таблицы нужны, какие атрибуты хранить.
  • Правильность связей — какие объекты как связаны (1:N, M:N), нужны ли промежуточные таблицы.
  • Корректность ограничений — бизнес-правила (дата возврата > дата выдачи, остаток >= 0) превращаются в CHECK и триггеры.
  • Эффективность запросов — понимание частых операций помогает правильно проиндексировать таблицы.
  • Безопасность — понимание ролей пользователей помогает настроить права доступа.

Вывод: плохой анализ приводит к неверной структуре → избыточность данных → аномалии → ошибки приложения → переработка всей системы.

29►Вопрос 29. Этапы проектирования БД. Логическое проектирование

Этапы проектирования

  • 1. Концептуальное проектирование — создание ER-диаграммы: определение сущностей, их атрибутов и связей. Не зависит от конкретной СУБД.
  • 2. Логическое проектирование — преобразование ER-диаграммы в реляционную схему: определение таблиц, первичных и внешних ключей, нормализация.
  • 3. Физическое проектирование — выбор СУБД, типов данных, создание индексов, партиционирование, настройка хранилища для оптимальной производительности.
  • 4. Реализация — создание объектов SQL (CREATE TABLE, CREATE INDEX) и загрузка данных.
  • 5. Тестирование и оптимизация — проверка корректности, производительности, нагрузочное тестирование.

Логическое проектирование

  • Каждая сущность ER-диаграммы → отдельная таблица.
  • Атрибуты сущности → столбцы таблицы с определёнными типами данных.
  • Ключевой атрибут → первичный ключ таблицы.
  • Связь 1:N — добавляется внешний ключ в таблицу на стороне «многие».
  • Связь M:N — создаётся промежуточная ассоциативная таблица с двумя внешними ключами.
  • Проводится нормализация до 3НФ.

Пример: Библиотека

char   name[50];
char   dept[50];
double salary;
};
int main() {
// Запись в бинарный файл
FILE *f = fopen("workers.dat", "wb");
Worker workers[] = {

{"Иванов И.И.", "Цех1", 150000.0},

{"Петров П.П.", "Цех2", 120000.0},

{"Сидоров С.С.", "Цех1", 165000.0},

{"Алиев А.А.", "Цех1", 140000.0}

};
fwrite(workers, sizeof(Worker), 4, f);
fclose(f);
// Чтение и расчёт по цеху
const char *targetDept = "Цех1";
FILE *fr = fopen("workers.dat", "rb");
Worker rw;
double totalSalary = 0.0;
int count = 0;
while (fread(&rw, sizeof(Worker), 1, fr)) {
if (strcmp(rw.dept, targetDept) == 0) {
totalSalary += rw.salary;
count++;
}
}
fclose(fr);
cout << "Цех: " << targetDept << endl;
cout << "Сотрудников: " << count << endl;
cout << "Общая сумма выплат: " << totalSalary << " тг." << endl;
if (count > 0)
cout << "Средняя зарплата: " << totalSalary / count << " тг." << endl;
return 0;
}
