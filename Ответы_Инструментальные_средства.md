# Ответы к ГАК: «Инструментальные средства разработки программ»

Формат: **Вопрос — Ответ**.

---

## II блок — теоретические вопросы

### Вопрос 1. Назначение и функции ИСрПО. Подбор средств для CI

**Инструментальные средства разработки ПО (ИСрПО)** — это программы, помогающие создавать,
тестировать, сопровождать и сопровождать другое ПО. Назначение — повысить
производительность разработчика, снизить число ошибок и автоматизировать рутинные операции.

**Основные функции:** редактирование кода, трансляция (компиляция/интерпретация), отладка,
тестирование, сборка, управление версиями, документирование, моделирование.

**Подбор средств для непрерывной интеграции (CI):**
- **Система контроля версий** — Git (GitHub/GitLab) — хранит код и историю изменений.
- **CI/CD-сервер** — Jenkins, GitLab CI, GitHub Actions — автоматически собирает и тестирует
  при каждом коммите.
- **Сборщик** — Maven/Gradle (Java), npm (JS), pip (Python).
- **Средства тестирования** — JUnit, pytest, Selenium.
- **Контейнеризация** — Docker — единое окружение для сборки и тестов.

**Обоснование выбора:** связка Git + GitHub Actions + Docker + автотесты обеспечивает, что
после каждого изменения код автоматически собирается, прогоняется через тесты и при успехе
готов к развёртыванию — это и есть суть непрерывной интеграции: раннее обнаружение ошибок и
стабильно работающая основная ветка.

### Вопрос 2. Методология, метод, средство. Классификация методологий

- **Методология** — система принципов, методов и правил организации процесса разработки ПО
  (что и в каком порядке делать).
- **Метод** — конкретный способ выполнения определённого этапа (например, метод декомпозиции,
  объектно-ориентированный анализ).
- **Средство (инструмент)** — программа или нотация, реализующая метод (IDE, CASE-средство,
  UML).

**Общая классификация методологий:**
- **Тяжёлые (предиктивные/каскадные)** — жёсткое планирование, подробная документация
  (Waterfall, RUP).
- **Гибкие (Agile)** — итеративность, адаптивность, минимум документации (Scrum, Kanban, XP).
- По подходу к проектированию: **структурные** (функциональная декомпозиция) и
  **объектно-ориентированные**.

### Вопрос 3. Модели ЖЦ ПО. Каскадная vs итеративная

**Жизненный цикл (ЖЦ) ПО** — период от идеи до снятия продукта с эксплуатации.

**Каскадная (водопадная) модель** — этапы выполняются строго последовательно (анализ →
проектирование → реализация → тестирование → внедрение), возврат к предыдущему этапу
затруднён. Подходит для проектов с чёткими, неизменными требованиями.

**Итеративная модель** — продукт создаётся за несколько повторяющихся циклов (итераций);
каждая итерация даёт работающую версию, требования уточняются по ходу.

| Критерий | Каскадная | Итеративная |
|----------|-----------|-------------|
| Порядок этапов | строго последовательный | повторяющиеся циклы |
| Требования | фиксированы в начале | уточняются по ходу |
| Результат | в конце проекта | после каждой итерации |
| Гибкость к изменениям | низкая | высокая |
| Риски | выявляются поздно | выявляются рано |

### Вопрос 4. Инструменты разработки vs инструменты сопровождения

- **Инструменты разработки** используются при создании продукта: редакторы кода,
  компиляторы, отладчики, IDE, средства проектирования (UML), сборщики. Цель — построить
  работающее ПО.
- **Инструменты сопровождения** используются после выпуска, при эксплуатации: системы учёта
  ошибок (баг-трекеры), мониторинг, профилировщики, средства управления версиями и
  обновлениями, средства резервного копирования. Цель — поддерживать и улучшать уже
  работающий продукт (исправление ошибок, адаптация, оптимизация).

Разработка создаёт продукт, сопровождение поддерживает его работоспособность и развитие на
протяжении эксплуатации.

### Вопрос 5. Гибкие методологии. Как Scrum и Kanban реализуют гибкий подход

**Гибкие методологии (Agile):** SCRUM, KANBAN, DSDM, MSF, ALM, XP, RAD — основаны на
итеративности, тесном взаимодействии с заказчиком, готовности к изменениям и быстрой
поставке работающих версий.

**SCRUM** реализует гибкость через короткие фиксированные итерации — **спринты** (1–4 недели).
Работа берётся из **бэклога продукта**, на спринт формируется **бэклог спринта**. Есть роли
(Product Owner, Scrum Master, команда), события (планирование, ежедневный стендап, обзор,
ретроспектива). После каждого спринта — работающий инкремент, что позволяет быстро
реагировать на изменения требований.

**KANBAN** реализует гибкость через визуализацию потока работ на **доске** (столбцы «To Do →
In Progress → Done») и **ограничение незавершённой работы (WIP-лимиты)**. Нет фиксированных
спринтов — задачи вытягиваются по мере освобождения исполнителей (поточная, непрерывная
поставка). Гибкость — в непрерывной приоритизации и быстром прохождении задач через поток.

Оба подхода итеративны и адаптивны, но Scrum — спринтами, Kanban — непрерывным потоком.

### Вопрос 6. Бизнес-моделирование. Типы моделей. Графическая vs имитационная

**Бизнес-моделирование** — описание бизнес-процессов организации для их анализа, оптимизации
и автоматизации.

**Типы моделей бизнес-процессов:**
- **Графические** — наглядное изображение процесса схемами/диаграммами (блок-схемы, IDEF, UML).
- **Имитационные** — модель, которая выполняется (симулируется) во времени и показывает
  поведение процесса при разных параметрах.
- **Исполняемые/функциональные** — формальное описание, которое можно автоматически
  исполнить (BPEL).
- **Информационные** — описывают данные и их потоки (ER-модели, DFD).

**Графическая vs имитационная:** графическая модель **статична** — это «чертёж» процесса для
понимания структуры. Имитационная модель **динамична** — её запускают, она воспроизводит
работу процесса во времени, позволяет оценить производительность, узкие места, влияние
изменений до их внедрения.

---

### Вопрос 7. Методы моделирования бизнес-процессов. Назначение Role Activity Diagram

**Методы:** Flow Chart Diagram (блок-схема), Data Flow Diagram (DFD — диаграмма потоков
данных), Role Activity Diagram (RAD — диаграмма ролей и действий), IDEF (IDEF0, IDEF3 и др.),
UML (Unified Modeling Language).

**Role Activity Diagram (RAD)** используется для описания бизнес-процесса с точки зрения
**ролей** (участников) и выполняемых ими **действий**, а также взаимодействия между ролями.
Она показывает, кто что делает, в каком порядке, где роли передают работу друг другу и где
происходит синхронизация/ветвление. Применяется, когда важно понять распределение
обязанностей и взаимодействие участников процесса.

### Вопрос 8. AllFusion Process Modeler и MS Visio

**AllFusion Process Modeler (ранее BPwin)** — специализированное CASE-средство для
функционального моделирования и анализа бизнес-процессов. Поддерживает нотации **IDEF0**
(функциональная модель), **IDEF3** (модель потоков работ) и **DFD**. Назначение — построение
структурированных моделей деятельности предприятия, анализ и оптимизация процессов.

**MS Visio** — универсальный редактор деловой графики и диаграмм от Microsoft. Позволяет
строить блок-схемы, схемы баз данных, UML-диаграммы, сетевые схемы, организационные
диаграммы. Назначение — наглядное визуальное документирование процессов, систем и структур;
менее формализован, чем BPwin, но гораздо шире по типам диаграмм.

Различие: AllFusion — строгий инструмент функционального моделирования (анализ), Visio —
универсальный инструмент рисования разнообразных схем (визуализация).

### Вопрос 9. Среды реализации. IDE. Цель использования. Популярность Visual Studio

**Интегрированная среда разработки (IDE)** объединяет в одной программе все инструменты
разработчика: редактор кода, компилятор/интерпретатор, отладчик, средства сборки и др.

**Основная цель использования среды разработки** — повысить производительность за счёт
объединения инструментов в едином интерфейсе: не нужно переключаться между разными
программами, всё взаимосвязано (например, ошибка компиляции сразу подсвечивается в редакторе).

**Почему Microsoft Visual Studio популярна:** мощная и зрелая IDE с поддержкой множества
языков (C#, C++, F#, VB и др.); богатый отладчик, IntelliSense (умное автодополнение),
интеграция с .NET, системами контроля версий, средствами тестирования и развёртывания;
большое сообщество, документация и расширения. Удобна для разработки от настольных до
веб- и облачных приложений.

### Вопрос 10. Виртуальные машины. Принцип работы. VirtualBox vs VMware Workstation

**Виртуальная машина (ВМ)** — программная эмуляция компьютера, позволяющая запускать внутри
одной физической машины другую (гостевую) ОС. **Принцип работы:** специальная программа —
**гипервизор** — выделяет виртуальные ресурсы (процессор, память, диск, сеть) и изолирует
гостевую ОС от хостовой, перехватывая и обрабатывая её обращения к оборудованию.

| Критерий | VirtualBox | VMware Workstation |
|----------|------------|--------------------|
| Разработчик | Oracle | VMware (Broadcom) |
| Лицензия | бесплатный, open source | платный (есть бесплатный Player) |
| Производительность | хорошая | выше, особенно с 3D/нагрузкой |
| Удобство/функции | базовые, достаточно для учёбы | продвинутые (снимки, клоны, интеграция) |
| Платформы | Windows, Linux, macOS | Windows, Linux |

VirtualBox — бесплатное универсальное решение для учёбы и личного использования; VMware
Workstation — профессиональный платный продукт с большей производительностью и
возможностями.

---

### Вопрос 11. ВМ: свойства и типы. Польза для тестирования ПО

**Свойства ВМ:** изоляция (гостевая ОС отделена от хоста), инкапсуляция (вся ВМ — это
набор файлов, легко копировать/переносить), независимость от оборудования, снимки состояния
(snapshots).

**Типы виртуализации/ВМ:**
- **Системные ВМ** — эмулируют целый компьютер с ОС (VirtualBox, VMware).
- **Программные (процессные) ВМ** — выполняют программы определённой среды (JVM для Java,
  CLR для .NET).
- По типу гипервизора: **Type 1** (нативный, работает прямо на железе — ESXi, Hyper-V) и
  **Type 2** (хостовый, поверх ОС — VirtualBox, VMware Workstation).

**Польза для тестирования ПО:** можно быстро создать чистое тестовое окружение, проверить
продукт на разных ОС и конфигурациях без отдельных компьютеров; снимки позволяют откатиться
к исходному состоянию после теста; изоляция защищает основную систему от сбоев и вредоносного
кода; легко воспроизводить условия и клонировать стенды.

### Вопрос 12. SDLC. Этапы технологии разработки. Определение

**SDLC (Software Development Life Cycle)** — жизненный цикл разработки ПО, совокупность
этапов от зарождения идеи до вывода продукта из эксплуатации.

**Этапы:**
1. **Анализ требований** — сбор и формализация требований заказчика.
2. **Проектирование** — разработка архитектуры и структуры системы.
3. **Реализация (кодирование)** — написание программного кода.
4. **Тестирование** — проверка соответствия требованиям, поиск ошибок.
5. **Внедрение (развёртывание)** — установка у заказчика, ввод в эксплуатацию.
6. **Сопровождение** — поддержка, исправление ошибок, доработка.

### Вопрос 13. ЖЦ ПО. Цель вспомогательных средств поддержки

**Вспомогательные средства поддержки ЖЦ** — это инструменты, которые сопровождают процесс
разработки, но не создают код напрямую. Их **цель** — обеспечить управление, контроль
качества и организацию работы на всех этапах ЖЦ.

К ним относятся: системы управления версиями (Git), системы управления проектами и задачами
(Jira, Trello), средства документирования, баг-трекеры, средства автоматизации сборки и
тестирования, средства резервного копирования.

Цель — повысить управляемость проекта, сохранить историю изменений, обеспечить
взаимодействие команды, контроль качества и прослеживаемость требований на протяжении всего
жизненного цикла.

### Вопрос 14. Спиральная модель ЖЦ

**Спиральная модель** (предложена Боэмом) сочетает итеративность с акцентом на **анализ
рисков**. Разработка идёт по виткам спирали; каждый виток включает четыре сектора:
1. **Определение целей** и требований к данному витку.
2. **Анализ рисков** и их снижение (главная особенность модели).
3. **Разработка и тестирование** очередной версии (прототипа).
4. **Планирование** следующего витка и оценка результатов заказчиком.

С каждым витком продукт уточняется и расширяется. Модель подходит для крупных, сложных и
рискованных проектов, где требования заранее не полностью ясны. Минус — сложность управления
и высокая стоимость анализа рисков.

### Вопрос 15. Отладка и тестирование: определения

- **Тестирование** — процесс выполнения программы с целью **обнаружить** наличие ошибок и
  проверить соответствие требованиям. Тестирование отвечает на вопрос «есть ли ошибка?».
- **Отладка (debugging)** — процесс **локализации (поиска) и устранения** уже обнаруженных
  ошибок. Отладка отвечает на вопрос «где ошибка и как её исправить?».

Тестирование выявляет факт ошибки, отладка находит её причину и исправляет.

---

### Вопрос 16. Этапы логического проектирования. Декомпозиция задачи

**Логическое проектирование** — этап, на котором определяется логическая структура будущей
программы независимо от конкретной реализации: алгоритмы, структуры данных, модули и связи
между ними.

**Декомпозиция задачи** — разбиение сложной исходной задачи на более простые подзадачи
(модули, функции), каждую из которых легче понять, реализовать и протестировать. Применяется
принцип «разделяй и властвуй»: задача делится сверху вниз до уровня элементарных операций.
Декомпозиция упрощает разработку, повышает читаемость, позволяет распределить работу между
исполнителями и повторно использовать модули.

### Вопрос 17. Программная среда разработки. Взаимодействие инструментов

**Программная среда разработки** — совокупность взаимосвязанных инструментов для создания ПО.
Взаимодействие инструментов: разработчик пишет код в **редакторе**; **компилятор/интерпретатор**
преобразует код в исполняемую форму, возвращая редактору сообщения об ошибках, которые сразу
подсвечиваются; **отладчик** позволяет пошагово выполнить программу, используя информацию
компилятора; **сборщик** объединяет модули и библиотеки в готовый продукт; **система контроля
версий** фиксирует изменения. Все компоненты обмениваются данными через единую среду, поэтому
результат работы одного инструмента сразу доступен другому — это и обеспечивает целостность и
скорость процесса.

### Вопрос 18. IDE и её структура. Классификация. Как облегчает программирование

**IDE (Integrated Development Environment)** — единая среда, объединяющая инструменты
разработки.

**Структура (компоненты):** редактор кода, компилятор/интерпретатор, отладчик, средства
автодополнения (IntelliSense), система сборки, интеграция с контролем версий, менеджер
проектов, средства тестирования.

**Классификация IDE:**
- по числу языков: **одноязычные** (Turbo Pascal) и **многоязычные** (Visual Studio, Eclipse);
- по платформе: **локальные** (настольные) и **облачные/онлайн** (Replit, Cloud9);
- по специализации: универсальные и предметно-ориентированные (например, для мобильной
  разработки — Android Studio).

**Как облегчает программирование:** объединяет все инструменты в одном окне, ускоряет ввод
кода (автодополнение, шаблоны), мгновенно показывает синтаксические ошибки, упрощает отладку
и сборку, автоматизирует рутину — это снижает число ошибок и повышает производительность.

### Вопрос 19. Выбор IDE. Инструменты, полезные при командной разработке

**Выбор IDE** зависит от языка, платформы, требований проекта, лицензии и привычек команды.

**Инструменты IDE, наиболее полезные при командной разработке:**
- **Интеграция с системой контроля версий (Git)** — совместная работа над кодом, история,
  ветвление, слияние.
- **Code review / комментирование** — просмотр изменений коллег.
- **Единый стиль кода (форматтеры, линтеры)** — согласованность кода в команде.
- **Средства совместной работы (Live Share)** — одновременное редактирование.
- **Интеграция с трекерами задач (Jira)** и **CI/CD** — связь кода с задачами и автосборкой.
- **Средства тестирования** — общий набор автотестов.

Эти инструменты обеспечивают согласованность, прозрачность и координацию работы участников.

### Вопрос 20. Обзор сред: JDK, BlackBox, Visual Studio, Eclipse. Роль IDE. Кроссплатформенность Eclipse

- **JDK (Java Development Kit)** — комплект разработчика Java (компилятор `javac`, JVM,
  библиотеки, утилиты). Строго говоря, это не IDE, а набор инструментов.
- **BlackBox** — среда разработки на языке Component Pascal (Oberon), компактная, учебная.
- **Microsoft Visual Studio** — мощная многоязычная IDE (преимущественно .NET, C#, C++).
- **Eclipse** — свободная расширяемая IDE, в первую очередь для Java, с плагинами для многих
  языков.

**Роль IDE в программировании:** объединяет редактор, компилятор, отладчик и другие средства,
автоматизирует разработку, ускоряет написание и отладку кода, снижает число ошибок.

**Почему Eclipse кроссплатформенна:** написана на Java и работает на JVM, поэтому запускается
на Windows, Linux, macOS без изменения кода; кроме того, имеет модульную (плагинную)
архитектуру, позволяющую добавлять поддержку разных языков и платформ.

### Вопрос 21. Windows SDK. Состав. Назначение

**SDK (Software Development Kit)** — комплект средств разработки для определённой платформы.

**Windows SDK** — набор инструментов для создания приложений под ОS Windows.

**Основные компоненты Windows SDK:**
- **Заголовочные файлы (headers)** и **библиотеки (libraries)** Windows API;
- **Компиляторы и компоновщики** (или интеграция с ними);
- **Документация и примеры** по Windows API;
- **Утилиты и инструменты** — отладчики, профилировщики, средства подписи и упаковки,
  редакторы ресурсов.

**Назначение SDK** при разработке Windows-приложений — предоставить разработчику всё
необходимое для обращения к функциям операционной системы (Windows API): создание окон,
работа с файлами, графикой, сетью и устройствами. Без SDK программист не имел бы интерфейсов
и инструментов для взаимодействия с платформой.

### Вопрос 22. ВМ: типы и свойства. Зачем в тестировании ПО

**Свойства ВМ:** изоляция, инкапсуляция (вся ВМ — набор файлов), аппаратная независимость,
поддержка снимков (snapshot), возможность клонирования.

**Типы:** системные (эмулируют компьютер целиком — VirtualBox, VMware) и процессные/программные
(JVM, .NET CLR); по гипервизору — Type 1 (нативные) и Type 2 (хостовые).

**Зачем в тестировании ПО:** ВМ позволяют тестировать программу на разных операционных
системах и конфигурациях без покупки нескольких компьютеров; обеспечивают чистое и
повторяемое окружение; снимки дают возможность мгновенно вернуть систему в исходное
состояние после теста; изоляция защищает основную ОС от сбоев и вирусов; легко создавать и
удалять тестовые стенды. Это ускоряет и удешевляет тестирование.

### Вопрос 23. Инструменты для работы с ВМ. Разница JDK и JVM

**Инструменты работы с ВМ:** VMware Workstation, Microsoft Virtual PC, Oracle VirtualBox,
Parallels Desktop (для macOS). Все они создают и управляют виртуальными машинами на
персональном компьютере.

**Разница JDK и JVM:**
- **JVM (Java Virtual Machine)** — виртуальная машина Java, которая **исполняет** байт-код
  Java; обеспечивает кроссплатформенность («написано один раз — работает везде»). Входит в
  состав среды выполнения JRE.
- **JDK (Java Development Kit)** — комплект **разработчика**: включает компилятор `javac`,
  библиотеки, утилиты **и** среду выполнения JRE (а значит и JVM).

Кратко: JDK нужен для **разработки** программ на Java и содержит JVM; JVM нужна для
**выполнения** уже скомпилированных программ. JDK ⊃ JRE ⊃ JVM.

### Вопрос 24. Правовые методы защиты программных продуктов и баз данных

ПО и базы данных охраняются как объекты интеллектуальной собственности.

**Правовые методы защиты:**
- **Авторское право** — защищает программу как литературное произведение (исходный и объектный
  код, базы данных как составные произведения); возникает автоматически при создании.
- **Патентное право** — защищает оригинальные технические решения, алгоритмы (в ряде стран).
- **Лицензионные соглашения (EULA)** — регулируют условия использования (проприетарные,
  свободные/открытые лицензии — GPL, MIT и т. д.).
- **Регистрация** программ и баз данных в уполномоченном органе — подтверждает авторство.
- **Коммерческая тайна** — защита нераскрытого кода и ноу-хау.
- **Товарный знак** — защита названия и бренда продукта.

Эти методы обеспечивают правообладателю исключительные права на использование,
распространение и защиту от незаконного копирования.

### Вопрос 25. Стандартизация в управлении качеством. Основные функции

**Стандартизация** в управлении качеством ПО — установление единых требований, правил и норм
к процессам и продуктам разработки (стандарты ISO/IEC, ГОСТ, ЕСПД).

**Основные функции стандартизации:**
- **Обеспечение качества** — единые критерии оценки продукта и процессов.
- **Унификация и совместимость** — согласованность форматов, интерфейсов, документации.
- **Регламентация процессов** — описание порядка разработки, тестирования, сопровождения.
- **Обеспечение надёжности и безопасности** ПО.
- **Упорядочение документации** (например, ЕСПД — единая система программной документации).
- **Повышение взаимозаменяемости и повторного использования** компонентов.
- **Основа для сертификации** продукта и процессов.

### Вопрос 26. Этапы физического проектирования ПО

**Физическое проектирование** — этап, на котором логическая модель преобразуется в конкретную
реализацию с учётом выбранной платформы, языка и СУБД.

**Основные этапы:**
1. **Выбор технических и программных средств** — язык, СУБД, платформа, архитектура.
2. **Проектирование физической структуры данных** — таблицы, индексы, типы данных,
   размещение на носителях.
3. **Проектирование модулей и интерфейсов** — конкретная структура программных компонентов.
4. **Распределение данных и компонентов** (для распределённых систем).
5. **Оптимизация** — производительность, объём памяти, скорость доступа.
6. **Подготовка к реализации** — спецификации для кодирования.

В отличие от логического (что делать), физическое проектирование определяет, как это будет
реализовано на конкретной платформе.

### Вопрос 27. Техническое задание. Разделы (пример)

**Техническое задание (ТЗ)** — основной документ, определяющий требования к разрабатываемому
ПО. По ГОСТ 19.201 (ЕСПД) типовые разделы:

1. **Введение** — наименование, область применения программы.
2. **Основания для разработки** — документ-основание, заказчик, исполнитель.
3. **Назначение разработки** — функциональное и эксплуатационное назначение.
4. **Требования к программе:**
   - требования к функциональным характеристикам;
   - требования к надёжности;
   - условия эксплуатации;
   - требования к составу и параметрам технических средств;
   - требования к информационной и программной совместимости;
   - требования к интерфейсу.
5. **Требования к программной документации.**
6. **Технико-экономические показатели.**
7. **Стадии и этапы разработки** — сроки, состав работ.
8. **Порядок контроля и приёмки** — виды испытаний, критерии приёмки.

### Вопрос 28. Чем отладка отличается от тестирования

| Критерий | Тестирование | Отладка |
|----------|--------------|---------|
| Цель | обнаружить наличие ошибок | найти причину и устранить ошибку |
| Вопрос | «есть ли ошибка?» | «где ошибка и как исправить?» |
| Когда | после написания кода, систематически | после того, как ошибка найдена |
| Кто чаще выполняет | тестировщики | разработчики |
| Результат | список найденных дефектов | исправленный код |
| Инструменты | тест-кейсы, автотесты, Selenium, JUnit | отладчики (debugger), точки останова |

Тестирование выявляет факт ошибки, отладка локализует её причину и исправляет код.

### Вопрос 29. CASE-средства. Средства моделирования процессов. Факторы эффективности

**CASE-средства (Computer-Aided Software Engineering)** — программы для автоматизации
проектирования и разработки ПО (моделирование, генерация кода, документирование).

**CASE-средства класса моделирования процессов:** AllFusion Process Modeler (BPwin),

**Основные факторы эффективности CASE-технологии:**

- повышение качества и сопровождаемости ПО;

Эффективность достигается только при правильном внедрении, обучении персонала и интеграции

### Вопрос 30. Трансляция, связывание/компоновка, загрузка

**Трансляция** — преобразование исходного кода в машинный (или промежуточный) код. Два вида:
- **Компиляция** — весь исходный код целиком переводится в исполняемый файл заранее
  (C, C++); выполняется быстро, но требует пересборки при изменениях.
- **Интерпретация** — код выполняется построчно интерпретатором во время работы (Python,
  JavaScript); гибче и переносимее, но медленнее.

**Тип связывания/компоновки (linking)** — объединение объектных модулей и библиотек в
исполняемую программу:
- **Статическое связывание** — код библиотек включается в исполняемый файл при сборке
  (файл больше, но автономен).
- **Динамическое связывание** — библиотеки (DLL/.so) подключаются во время выполнения
  (файл меньше, библиотеки общие, но нужна их наличие при запуске).

**Загрузка (loading)** — размещение готовой программы и её библиотек в оперативной памяти и
передача ей управления для выполнения; этим занимается загрузчик ОС.

Общий путь: исходный код → (трансляция) объектный код → (компоновка) исполняемый файл →
(загрузка) выполнение в памяти.

### Задача 2. Сравнение моделей жизненного цикла

| Модель | Принцип | Гибкость | Когда применять | Пример/инструмент |
|--------|---------|----------|-----------------|-------------------|
| **Каскадная (Waterfall)** | строгая последовательность этапов | низкая | чёткие неизменные требования | гос. проекты, ГОСТ-разработка |
| **Итеративная** | повторяющиеся циклы, нарастающий продукт | средняя | требования уточняются | RUP |

| **Гибкая (Agile/Scrum)** | короткие спринты, адаптивность | очень высокая | меняющиеся требования | Scrum, Kanban |

### Задача 3. Концептуальная модель (этап логического проектирования)

**Предметная область: «Библиотека».** На этапе логического проектирования при системном
подходе выделяют сущности предметной области, их атрибуты и связи.

**Концептуальная модель (сущности и связи):**

```
[ЧИТАТЕЛЬ] ──(берёт)──< [ВЫДАЧА] >──(содержит)── [КНИГА] ──(относится к)──< [АВТОР]
   │                        │                        │
 № билета               дата выдачи              инв. номер
 ФИО                    дата возврата            название
 адрес                                           год издания
```

**Сущности:**
- **Читатель** (№ билета, ФИО, адрес);
- **Книга** (инв. номер, название, год издания);
- **Автор** (код, имя);
- **Выдача** (дата выдачи, дата возврата) — связывает читателя и книгу.

**Связи:** читатель может взять много книг; книга может быть выдана многим читателям в разное
время (связь «многие-ко-многим» через сущность «Выдача»); у книги один или несколько авторов.

Концептуальная модель не зависит от конкретной СУБД и описывает суть предметной области.

### Задача 5. Моделирование бизнес-процесса в UML (Use Case + Activity)

**Бизнес-процесс: «Покупка билета в кинотеатре».**

**Диаграмма вариантов использования (Use Case):**

```
                ┌──────────────────────────────────────┐
                │           Система кинотеатра          │
                │   ( Просмотр сеансов )                │
  ┌──────┐      │   ( Выбор места )                     │      ┌──────────┐
  │Клиент│──────│   ( Оплата билета )                   │──────│ Кассир   │
  └──────┘      │   ( Печать билета )                   │      └──────────┘
                └──────────────────────────────────────┘
```

**Диаграмма активности (Activity):**

```
 (start)
    ▼
[Выбрать фильм и сеанс]
    ▼
[Выбрать свободное место]
    ▼
 ◇ Место свободно? ◇── нет ──► [Выбрать другое место] ─┐
    │ да                                                │
    ▼ ◄──────────────────────────────────────────────┘
[Оплатить билет]
    ▼
[Получить билет]
    ▼
 (end)
```

│   СТУДЕНТ     │        │     ИЗУЧАЕТ       │        │  ДИСЦИПЛИНА  │
├──────────────┤  1   N ├──────────────────┤ N   1  ├──────────────┤
│ id_студента ●│───────<│ id_студента (FK) │>───────│ id_дисц.    ●│
│ ФИО          │        │ id_дисц.   (FK)  │        │ название     │

```

**Описание:** сущности **Студент** и **Дисциплина** связаны отношением «многие-ко-многим»
(студент изучает много дисциплин, дисциплину изучают многие студенты), которое
разрешается через промежуточную сущность **Изучает** (с атрибутами «оценка», «дата»).
● — первичный ключ, FK — внешний ключ.

**Применение:** ER-диаграммы используются при проектировании **баз данных** (предметная

### Задача 9. SDK и API: написание простого приложения

**Решение: Windows-приложение на C# с использованием .NET SDK и Windows Forms API.**

**Процесс написания (форма с кнопкой и приветствием):**

```csharp
using System;
using System.Windows.Forms;   // используем API Windows Forms из SDK

namespace HelloApp
{
    public class MainForm : Form
    {
        public MainForm()
        {
            Text = "Пример SDK/API";

**Пояснение процесса:** SDK (.NET) предоставляет компилятор и библиотеки; через **API**

### Задача 11. Библиотеки, компоновщик, связывание, трансляция

**Виды библиотек при создании исполняемого файла:**
- **Статические библиотеки** (`.lib`, `.a`) — их код **включается** в исполняемый файл на
  этапе компоновки.
- **Динамические библиотеки** (`.dll` в Windows, `.so` в Linux) — подключаются во время
  выполнения, не входят в сам exe.

**Структура создания исполняемого файла:**

```
 исходный код (.c)
        │  компиляция
        ▼
 объектный код (.obj/.o)  +  библиотеки (.lib/.dll)
        │  компоновка (linker)
        ▼
 исполняемый файл (.exe)
        │  загрузка (loader)
        ▼
 выполнение в памяти
```

**Сравнение типов связывания:**

| | Статическое | Динамическое |
|--|-------------|--------------|
| Когда подключается | при сборке | при выполнении |
| Размер exe | больше | меньше |
| Автономность | полная | нужна DLL |
| Обновление библиотеки | пересборка | замена DLL |

**Трансляция — пример (Python, интерпретация):**

```python
# greet.py — интерпретатор выполняет код построчно
name = input("Имя: ")
print("Привет,", name)
```

**Компиляция — пример (C):**

```c
/* greet.c — компилятор создаёт исполняемый файл целиком */
#include <stdio.h>
int main() {
    char name[50];
    printf("Имя: ");
    scanf("%49s", name);
    printf("Привет, %s\n", name);
    return 0;
}
/* Сборка: gcc greet.c -o greet   →   запуск: ./greet */
```

Компиляция переводит весь код заранее в `.exe`; интерпретация выполняет код построчно во
время работы.

### Задача 12. Структура процесса отладки (блок-схема) + код + метод тестирования

**Блок-схема процесса отладки:**

```
        ┌─────────┐
        │  Начало │
        └────┬────┘
             ▼
   ┌────────────────────┐
   │ Запуск программы /  │
   │ выполнение теста    │
   └────────┬───────────┘
            ▼
        ◇ Есть ошибка? ◇──── Нет ──► ┌──────────┐
            │ Да                     │  Конец   │
            ▼                        └──────────┘
   ┌────────────────────┐
   │ Локализация ошибки  │ (точки останова,
   │ (поиск места)       │  просмотр переменных)
   └────────┬───────────┘
            ▼
   ┌────────────────────┐
   │ Исправление кода    │
   └────────┬───────────┘
            ▼
   ┌────────────────────┐
   │ Повторный запуск    │
   └────────┬───────────┘
            └──────► (назад к проверке «Есть ошибка?»)
```

**Фрагмент кода с ошибкой и отладкой:**

```python
def average(numbers):
    total = 0
    for n in numbers:
        total += n
    return total / len(numbers)   # ошибка: деление на ноль при пустом списке

# Отладка: ставим точку останова, проверяем len(numbers).
# Исправление — проверка на пустой список:
def average_fixed(numbers):
    if not numbers:
        return 0
    return sum(numbers) / len(numbers)

print(average_fixed([4, 6, 8]))   # 6.0
print(average_fixed([]))          # 0
```

**Работа с инструментом отладки:** в IDE ставится **точка останова (breakpoint)** на нужной
строке, программа запускается в режиме отладки, выполнение приостанавливается, и можно
посмотреть значения переменных, выполнить код пошагово (Step Over/Into), найти строку, где
значение становится некорректным.

**Метод тестирования: модульное (unit) тестирование** — пример ситуации: проверяем функцию
`average_fixed` на разных входных данных (обычный список, пустой список, один элемент),
сравнивая результат с ожидаемым:

```python
assert average_fixed([2, 4]) == 3.0
assert average_fixed([]) == 0
```

Модульное тестирование применяют, чтобы убедиться в корректной работе отдельной функции в
граничных и обычных случаях.
